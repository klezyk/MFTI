# Прикладное применение: линейные модели и методы оптимизации

> Этот документ дополняет теоретический конспект (SUMMARY.md) практической перспективой: где и как методы из урока 6.1 применяются в реальных задачах IT, аналитики данных, финансов и поиска аномалий.

---

## 1. Логистическая регрессия

### Где применяется

**Кредитный скоринг (финансы).** Логистическая регрессия — золотой стандарт в банковском скоринге. Модель принимает на вход признаки заёмщика (доход, стаж, кредитная история, количество открытых счетов) и выдаёт вероятность дефолта. Банки любят её за интерпретируемость: регулятор (ЦБ, Basel Committee) требует объяснить, *почему* клиенту отказано. У логрегрессии каждый коэффициент — это вклад конкретного признака, что позволяет сказать: «отказ из-за высокой долговой нагрузки (вес +2.3) и короткого стажа на текущем месте (вес +1.1)». Ни одна «чёрная коробка» (нейросеть, gradient boosting) не даёт такой прозрачности из коробки.

**Fraud detection (финансы, IT).** Определение мошеннических транзакций в реальном времени. Когда карточная транзакция проходит через платёжный шлюз, модель за миллисекунды оценивает вероятность фрода по признакам: сумма, время суток, геолокация, тип мерчанта, расстояние от предыдущей транзакции. Логрегрессия здесь выигрывает за счёт скорости inference — это линейная операция (умножение вектора на матрицу), которая выполняется за микросекунды даже на слабом железе. В системах с миллионами транзакций в секунду это критично.

**Churn prediction (IT, телеком).** Предсказание оттока клиентов: уйдёт ли пользователь в следующем месяце? Признаки — активность за последние 30/60/90 дней, количество обращений в поддержку, тарифный план, длительность подписки. Выход — вероятность от 0 до 1, по которой можно ранжировать клиентов и направлять ресурсы retention-команды на тех, кто «на грани».

**A/B-тестирование и uplift-моделирование.** Логрегрессия используется для оценки статистической значимости различий между контрольной и тестовой группой с поправкой на ковариаты (возраст, регион, платформа). Это более мощный инструмент, чем простой t-тест, когда нужно учесть влияние confounders.

### Где НЕ стоит применять

- **Нелинейные зависимости.** Если граница между классами — не гиперплоскость (например, «аномальные» значения находятся в центре облака нормальных данных), логрегрессия провалится. Классический пример — задача XOR.
- **Изображения, текст, аудио.** Пиксели и токены требуют извлечения сложных иерархических признаков — это территория нейросетей. Логрегрессию можно использовать *поверх* эмбеддингов, но не напрямую на сырых данных (хотя в задании 6.1 она работает на MNIST — это исключение, потому что цифры 0 и 1 линейно разделимы).
- **Сложные взаимодействия признаков.** Если результат зависит от комбинации «признак A высокий И признак B низкий», линейная модель этого не поймает без ручного feature engineering (добавления произведений признаков).

### Типичные ошибки

1. **Забыть про калибровку.** Логрегрессия выдаёт *калиброванные* вероятности (в отличие от SVM или Random Forest). Но только при условии, что данные не сильно несбалансированы. Если фрод — 0.1% данных, то предсказанные вероятности будут занижены без коррекции.
2. **Интерпретировать коэффициенты без стандартизации.** Коэффициент 0.5 у «дохода в рублях» и 3.0 у «количества детей» не значит, что дети важнее — масштабы разные. Сравнивать веса можно только после стандартизации.
3. **Игнорировать мультиколлинеарность.** Два сильно коррелирующих признака (например, «доход» и «расходы по карте») делают коэффициенты нестабильными — знаки могут даже поменяться. Решение: L2-регуляризация или удаление одного из признаков.

## 2. Регуляризация: L1, L2, Elastic Net

### L1-регуляризация (Lasso)

**Суть для практика.** L1 обнуляет веса «бесполезных» признаков, превращая модель в автоматический feature selector. Это не просто удобство — это принципиально меняет workflow.

**Где применяется:**

- **Геномика и биоинформатика.** Тысячи генов, из которых значимы десятки. L1 находит «иголки в стоге сена», отбрасывая шум. Аналогичная ситуация в аналитике данных, когда из сотен метрик нужно выбрать ключевые.
- **Анализ логов и мониторинг (IT).** Система собирает 500 метрик (CPU, memory, disk I/O, network, latency по каждому эндпоинту...). Какие из них предсказывают инцидент? L1-регуляризованная модель оставит 10-15 значимых, остальные обнулит. Это даёт понимание, что мониторить в первую очередь.
- **Маркетинговая аналитика.** Из десятков каналов привлечения (SEO, контекст, email, push, ретаргетинг, партнёрки) L1 покажет, какие реально влияют на конверсию, а какие — шум.

**Когда НЕ подходит:**

- Когда есть группы коррелирующих признаков (L1 выберет один произвольный из группы, а остальные обнулит — вы потеряете информацию о том, что вся группа важна).
- Когда все признаки потенциально значимы и вам не нужна разреженность.

### L2-регуляризация (Ridge)

**Суть для практика.** L2 не обнуляет веса, а «сжимает» их к нулю. Это стабилизирует модель, особенно когда признаков много или они коррелируют.

**Где применяется:**

- **Финансовое моделирование.** Макроэкономические показатели сильно коррелируют (ВВП, инфляция, ставка ЦБ, безработица). Ridge не выкинет ни один из них, а даст стабильные оценки для всех — это важно для интерпретации и стресс-тестирования.
- **Рекомендательные системы.** В коллаборативной фильтрации L2 предотвращает переобучение на редких пользователях с 1-2 оценками: без регуляризации модель запомнит их оценки «наизусть», с L2 — обобщит.
- **NLP: классификация текста.** TF-IDF-вектор для документа содержит тысячи компонент (по количеству слов в словаре). Ridge стабилизирует модель, не отбрасывая слова — каждое слово вносит маленький, но ненулевой вклад.

**Когда НЕ подходит:**

- Когда вам нужна разреженная модель (для интерпретации или ускорения inference). Ridge никогда не обнулит коэффициент полностью.

### Elastic Net (L1 + L2)

**Суть для практика.** Комбинация: L1 отбирает признаки, L2 стабилизирует оставшиеся. Лучшее из двух миров.

**Где применяется:**

- **Поиск аномалий с высокоразмерными данными.** Логи серверов, данные сенсоров, финансовые транзакции — часто 100+ признаков, из которых многие коррелируют (например, request_time и response_time). Elastic Net выберет информативное подмножество, но не «сломается» на коррелирующих группах.
- **Кредитный скоринг (продвинутый).** Банк собирает 200+ признаков о заёмщике. Часть из них дублируют друг друга (доход_заявленный, доход_из_ПФР, доход_по_карте). Elastic Net выберет одного «представителя» из каждой группы и стабильно оценит его вес.
- **Предиктивное обслуживание (IT/IoT).** Предсказание отказа оборудования по данным датчиков. Датчики часто коррелируют (температура корпуса и температура процессора), а Elastic Net корректно работает с такими данными.

### Сравнительная таблица: выбор регуляризации

| Ситуация | L1 (Lasso) | L2 (Ridge) | Elastic Net |
|----------|:----------:|:----------:|:-----------:|
| Много бесполезных признаков | **+++** | + | ++ |
| Коррелирующие признаки | — | **+++** | ++ |
| Нужна интерпретация (какие признаки важны) | **+++** | + | ++ |
| Нужна стабильность модели | + | **+++** | ++ |
| Мало данных, много признаков | ++ | ++ | **+++** |
| Все признаки потенциально полезны | — | **+++** | + |

**Правило большого пальца:** начни с Elastic Net — он работает в большинстве ситуаций. Если нужна максимальная разреженность — переключись на L1. Если все признаки важны — на L2.

## 3. Градиентный спуск и SGD

### Зачем понимать оптимизацию, если sklearn делает всё за тебя

Понимание того, как работает SGD, критично в трёх ситуациях:

1. **Большие данные.** Когда датасет не помещается в память (миллиарды транзакций, логов, событий), полный gradient descent невозможен — нужен SGD или mini-batch SGD. Понимание их особенностей позволяет правильно настроить обучение.
2. **Онлайн-обучение.** Fraud detection в реальном времени: модель обновляется на каждой новой транзакции (или батче), не переобучаясь на всех исторических данных. Это по сути SGD в production.
3. **Deep Learning.** Все нейросети обучаются вариантами SGD (Adam, AdaGrad, RMSProp). Понимание базового SGD — фундамент для работы с любыми нейросетями.

### Batch vs Mini-batch vs Стохастический: когда что

**Full Batch Gradient Descent:**
- Использует весь датасет для каждого шага
- Стабильная сходимость, но медленная
- Подходит: маленькие датасеты (до ~100K строк), когда нужна точная оптимизация
- Пример: обучение скоринговой модели на исторических данных о 50K заёмщиках

**Mini-batch SGD:**
- Использует случайные подмножества (128-1024 примера)
- Баланс скорости и стабильности
- Подходит: большинство задач, датасеты от 100K до миллиардов строк
- Пример: обучение модели fraud detection на 100M транзакций — batch_size=512 позволяет обработать весь датасет за разумное время

**Стохастический (batch_size=1):**
- Каждый шаг по одному примеру
- Шумный, но быстро покидает локальные минимумы
- Подходит: онлайн-обучение, потоковые данные
- Пример: обновление модели рекомендаций после каждого клика пользователя

### Learning rate на практике

**Слишком маленький learning rate** — модель сходится, но обучение занимает часы вместо минут. В production это означает, что модель не успевает переобучиться до следующего релиза.

**Слишком большой learning rate** — модель «прыгает» вокруг оптимума и никогда не сходится. На графике loss — хаотичные скачки вместо плавного снижения.

**На практике:** в sklearn `LogisticRegression` learning rate подбирается автоматически (solver='lbfgs'). Но если вы используете `SGDClassifier` или обучаете нейросеть, типичная стратегия:
- Начать с lr=0.01
- Если loss скачет — уменьшить в 10 раз
- Если loss падает слишком медленно — увеличить в 3-5 раз
- Использовать расписание (learning rate schedule): высокий lr в начале, постепенное снижение к концу обучения

### Типичные проблемы в production

- **Gradient explosion.** Если данные не стандартизированы, градиенты могут быть огромными (например, пиксели 0-255 в задании 6.1), что приводит к расхождению. Решение: StandardScaler *перед* обучением.
- **Стагнация на плато.** Loss перестал падать, но модель ещё не сошлась. Причина часто в слишком малом lr или в том, что модель уже нашла оптимум для данной архитектуры.
- **Воспроизводимость.** SGD использует случайный порядок данных, поэтому разные запуски дают разные модели. В production это проблема: две одинаковые модели предсказывают разное. Решение: фиксировать `random_state=42` (как в задании) или обучать ансамбль.

## 4. Стандартизация и предобработка

### Почему стандартизация критична для линейных моделей

Линейная модель считает `w1*x1 + w2*x2 + ...`. Если x1 — доход в рублях (30 000-500 000), а x2 — возраст (18-70), то градиент по x1 будет в тысячи раз больше, чем по x2. Модель будет оптимизировать в основном вес дохода, игнорируя возраст. Стандартизация (вычесть среднее, разделить на стандартное отклонение) приводит все признаки к одному масштабу.

**Важно:** древесные модели (Random Forest, XGBoost) НЕ требуют стандартизации — они работают с порядком значений, а не с абсолютными величинами. Это одно из преимуществ деревьев.

### Pipeline: защита от data leakage

**Проблема.** Вы стандартизируете *весь* датасет (train + test), потом делите. Модель «видела» статистику тестовых данных через среднее и дисперсию. Метрики на тесте завышены — в production модель работает хуже.

**Решение.** Pipeline: `make_pipeline(StandardScaler(), LogisticRegression())`. Scaler обучается *только на train*, затем применяется к test с теми же параметрами. При кросс-валидации это происходит автоматически для каждого фолда.

**В production** это особенно критично: scaler, обученный на исторических данных, применяется к новым транзакциям. Если его неправильно сохранить или загрузить — модель будет предсказывать мусор.

### Типичные ошибки при деплое

1. **Забыть сохранить scaler вместе с моделью.** Сохранили модель, забыли scaler. Новые данные подаются нестандартизированными — предсказания бессмысленны.
2. **Переобучить scaler на новых данных.** Scaler должен использовать статистику *обучающей* выборки. Если пересчитать mean/std на новых данных, предсказания будут несопоставимы с обучением.
3. **Разная предобработка для train и inference.** Классика: на обучении использовался one-hot encoding с `drop_first=True`, а в production — без. Или порядок категорий изменился.

## 5. Кросс-валидация и оценка моделей

### Зачем в реальных проектах

**Простой train/test split ненадёжен.** Если данных мало (что часто бывает в финансах — дефолтов мало, фродов мало), один «неудачный» split может завысить или занизить метрику. Кросс-валидация (CV-5, CV-10) даёт *распределение* метрики: вместо «accuracy=0.82» вы получаете «accuracy=0.80+-0.03». Это разница между «модель хорошая» и «модель может быть хорошей, а может — нет».

### Стратифицированная CV для несбалансированных данных

В задачах fraud detection и поиска аномалий положительный класс — 0.01-1% данных. При обычном k-fold split может случиться, что в одном фолде все фроды попали в train, а в test — ни одного. Стратифицированная CV (`StratifiedKFold`) гарантирует, что пропорция классов сохраняется в каждом фолде. В sklearn `cross_val_score` для классификации использует стратификацию по умолчанию.

### Метрики beyond accuracy

**Accuracy вводит в заблуждение на несбалансированных данных.** Если фрод — 0.1% транзакций, модель «всегда говори нет» даёт accuracy=99.9%. Но она бесполезна.

Что использовать вместо accuracy:

| Метрика | Когда использовать | Пример |
|---------|-------------------|--------|
| **Precision** | Когда цена false positive высока | Блокировка легитимной транзакции = потеря клиента |
| **Recall** | Когда цена false negative высока | Пропущенный фрод = финансовые потери |
| **F1-score** | Когда важен баланс precision и recall | Churn prediction: хотим найти больше «уходящих», но не задавить retention-команду ложными сигналами |
| **AUC-ROC** | Для общего сравнения моделей | Сравнение логрегрессии и Random Forest на одних данных |
| **AUC-PR** | При сильном дисбалансе классов | Fraud detection с 0.01% положительных примеров |

**Для поиска аномалий:** AUC-PR обычно информативнее, чем AUC-ROC, потому что ROC может завышать качество модели при большом дисбалансе классов (кривая выглядит хорошо за счёт огромного количества true negatives).

## 6. Сигмоида и кросс-энтропия: практический смысл

### Сигмоида — не просто активация

Сигмоида превращает произвольное число в вероятность (от 0 до 1). Это не просто математический трюк — это позволяет:

- **Настраивать порог принятия решения.** По умолчанию порог 0.5, но в fraud detection лучше поставить 0.1 (ловим больше фродов за счёт false positives). А при фильтрации спама — 0.9 (лучше пропустить спам, чем заблокировать легитимное письмо).
- **Ранжировать по вероятности.** Retention-команда обрабатывает клиентов в порядке убывания вероятности churn. Модель не просто говорит «уйдёт/останется», а даёт приоритет: 0.92, 0.87, 0.73...
- **Калибровать модель.** Если модель говорит «вероятность фрода 0.7», то из 100 таких транзакций действительно ~70 должны быть фродом. Это важно для оценки рисков и принятия бизнес-решений.

### Кросс-энтропия — не просто loss

Кросс-энтропия штрафует «уверенные ошибки» экспоненциально сильнее, чем «неуверенные». Если модель предсказала p=0.99 для фрода, а это легитимная транзакция — штраф огромный. Если предсказала p=0.55 — штраф маленький. Это заставляет модель быть честной в своей уверенности, что критично для систем, где на вероятности строятся бизнес-решения.

## 7. Линейные vs нелинейные модели: когда что выбирать

### Сравнительная таблица

| Критерий | Логистическая регрессия | Random Forest | XGBoost / LightGBM |
|----------|:----------------------:|:-------------:|:-------------------:|
| **Скорость обучения** | Очень быстрая | Средняя | Средняя |
| **Скорость inference** | Микросекунды | Миллисекунды | Миллисекунды |
| **Интерпретируемость** | Высокая (коэффициенты) | Средняя (feature importance) | Средняя (SHAP) |
| **Нелинейные зависимости** | Не ловит | Ловит | Ловит отлично |
| **Взаимодействия признаков** | Только вручную | Автоматически | Автоматически |
| **Требует стандартизации** | Да | Нет | Нет |
| **Устойчивость к выбросам** | Низкая | Высокая | Средняя |
| **Малый датасет (<1K)** | Хороша | Переобучение | Переобучение |
| **Регуляторные требования** | Полное соответствие | Сложнее объяснить | Сложнее объяснить |

### Когда линейные модели побеждают

1. **Регулируемые отрасли.** Банки, страхование, медицина — регулятор требует объяснимость. Логрегрессия — выбор по умолчанию.
2. **Real-time с жёсткими SLA.** Если inference должен занимать <100 мкс (платёжные системы, высокочастотный трейдинг) — линейная модель вне конкуренции.
3. **Мало данных.** На 500 примерах логрегрессия с L2 обычно стабильнее, чем Random Forest, который склонен к переобучению.
4. **Много признаков, мало примеров (p >> n).** Текстовая классификация с bag-of-words: 50K слов, 1K документов. L1-логрегрессия работает, деревья — нет.
5. **Baseline.** Всегда стоит начинать с логрегрессии, чтобы понять, какое качество можно получить «бесплатно». Если логрегрессия даёт AUC=0.95, сложные модели дадут 0.96 — стоит ли оно того?

### Когда лучше выбрать нелинейные модели

1. **Табличные данные со сложными зависимостями.** Если зависимость нелинейная (например, риск растёт при доходе <30K и >500K, но не посередине) — XGBoost поймает это автоматически.
2. **Категориальные признаки с высокой кардинальностью.** Город (1000 значений), user_agent (500 значений) — деревья обработают нативно, логрегрессии потребуется one-hot и регуляризация.
3. **Максимизация метрики в соревнованиях.** На Kaggle почти всегда побеждает gradient boosting. Но Kaggle — это не production.

### Фреймворк выбора модели

```
1. Есть ли регуляторные требования к объяснимости?
   -> Да -> Логистическая регрессия

2. Нужен ли inference <1 мс?
   -> Да -> Логистическая регрессия

3. Данных меньше 1000?
   -> Да -> Логистическая регрессия с L2

4. Признаков больше, чем примеров?
   -> Да -> Логистическая регрессия с L1 / Elastic Net

5. Во всех остальных случаях:
   -> Начать с логрегрессии (baseline)
   -> Попробовать gradient boosting (XGBoost / LightGBM)
   -> Сравнить метрики на кросс-валидации
   -> Выбрать с учётом trade-off: качество vs интерпретируемость vs скорость
```

## 8. Чек-лист практика

| Задача | Рекомендуемый подход |
|--------|---------------------|
| Кредитный скоринг | Логрегрессия + L2, Pipeline со StandardScaler, стратифицированная CV |
| Fraud detection (real-time) | Логрегрессия (скорость), порог 0.1-0.3, AUC-PR как метрика |
| Fraud detection (batch) | XGBoost как основная модель, логрегрессия как baseline |
| Churn prediction | Логрегрессия (baseline) -> XGBoost, F1/AUC-ROC для сравнения |
| Отбор значимых метрик мониторинга | L1-регуляризация — автоматически обнулит неважные |
| Аномалии в логах (100+ метрик) | Elastic Net (отбор + стабильность при корреляциях) |
| Предиктивное обслуживание | Elastic Net -> gradient boosting, Pipeline обязателен |
| Рекомендации (коллаборативная фильтрация) | L2-регуляризация для стабильности |
| Текстовая классификация (TF-IDF) | L2-логрегрессия (p >> n) |
| Онлайн-обучение на потоке данных | SGDClassifier с mini-batch |
| Маленький датасет (<1K строк) | Логрегрессия + L2 + CV-10 |
| Максимум качества без ограничений | XGBoost/LightGBM с подбором гиперпараметров |
