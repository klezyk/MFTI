## Ключевые концепции

### Задача классификации
Классификация — это задача машинного обучения с учителем (supervised learning), в которой модель предсказывает **категорию** (класс) для каждого объекта. В нашем случае — бинарная классификация: жив персонаж (`isAlive=1`) или мёртв (`isAlive=0`). Качество оценивается метрикой **Accuracy** — долей правильных ответов.

### Метрика Accuracy
$$\text{Accuracy} = \frac{\text{Число верных предсказаний}}{\text{Общее число предсказаний}}$$

Простая и интуитивная метрика, но может быть обманчивой при дисбалансе классов. Например, если 78% персонажей живы, модель-«заглушка», предсказывающая всем `isAlive=1`, получит Accuracy = 0.78 — при этом не уловив ни одного мёртвого персонажа.

### Data Leakage (утечка данных)
Утечка данных — ситуация, когда в признаки попадает информация, которая в реальности была бы недоступна на момент предсказания. Например, признак `dateOfDeath` напрямую выдаёт ответ: если дата смерти заполнена — персонаж мёртв. Такой признак обязательно удаляется, иначе модель покажет нереалистично высокое качество на обучении, но провалится на новых данных.

### Пропущенные значения (NaN)
Пропуски в данных — типичная ситуация в реальных датасетах. Удалять строки с NaN (`dropna()`) можно только если пропусков мало. Когда пропусков 80%+ (как `age` или `dateOfBirth`), лучше создать **два признака**: значение (с заменой NaN на 0) и индикатор пропуска (0/1). Так модель получает информацию и о значении, и о факте его наличия.

### Кросс-валидация (Cross-Validation)
Кросс-валидация — метод оценки модели, при котором данные разбиваются на K частей (фолдов). Модель K раз обучается на K-1 частях и тестируется на оставшейся. Результат — среднее качество по всем фолдам. Это даёт более надёжную оценку, чем однократное разделение на train/val, потому что каждый объект хотя бы раз побывает в тестовой части.

### Feature Engineering (конструирование признаков)
Создание новых признаков из существующих — часто более эффективный способ повысить качество модели, чем усложнение самой модели. Примеры из задания: логарифмирование `popularity` (борьба со скошенным распределением), бинаризация `numDeadRelations` (есть мёртвые родственники или нет), группировка культур в крупные регионы.

### Pipeline (конвейер обработки)
`Pipeline` из sklearn объединяет шаги предобработки и модель в единый объект. Это гарантирует, что одни и те же трансформации применяются к train и test, и предотвращает утечку данных при кросс-валидации (например, `StandardScaler` обучается только на train-фолде).

### One-Hot Encoding (горячее кодирование)
Способ преобразования категориального признака в числовой формат: для каждого уникального значения создаётся отдельная бинарная колонка (0/1). Например, `culture_grouped` с 12 категориями (включая `culture_no_data`) превращается в 12 колонок. Необходим для моделей, которые не умеют работать с текстовыми признаками.

### Масштабирование признаков (StandardScaler)
Приведение признаков к единому масштабу (среднее=0, std=1). Критически важно для моделей, чувствительных к масштабу: логистическая регрессия, KNN, SVM. Для деревьев и ансамблей на основе деревьев масштабирование не требуется, так как они работают с порогами на отдельных признаках.

## Данные и предобработка

### Описание датасета
- **Источник**: персонажи вселенной «Игра Престолов» из вики A Wiki of Ice and Fire
- **Размер**: 1557 объектов (train), 389 объектов (test)
- **Целевая переменная**: `isAlive` (1 — жив, 0 — мёртв)
- **Баланс классов**: ~78% живых, ~22% мёртвых (дисбаланс)

### Основные признаки
- `name`, `title`, `house`, `culture` — текстовые/категориальные
- `male`, `isMarried`, `isNoble` — бинарные
- `book1`-`book5` — появление в книгах (бинарные)
- `popularity` — числовой (0-1), количество связей персонажа
- `age`, `dateOfBirth` — числовые с 82% пропусков
- `numDeadRelations` — количество умерших родственников
- `isAliveMother/Father/Heir/Spouse` — бинарные с пропусками (87-98%)

### Что было сделано и зачем

| Шаг | Что | Зачем |
|-----|-----|-------|
| Удаление `dateOfDeath` | Убрали признак, дублирующий таргет | Предотвращение data leakage |
| Удаление `name`, `title`, `house`, `spouse`, `father`, `mother`, `heir` | Текстовые признаки с высокой кардинальностью | Модели не могут работать с произвольным текстом; 315 уникальных домов невозможно нормально закодировать |
| Удаление `isAliveMother/Father/Heir` | 98%+ пропусков | Слишком мало данных для извлечения сигнала |
| `popularity` -> `log10(popularity * 100 + 1)` | Логарифмическое преобразование | Распределение сильно скошено вправо; лог делает его ближе к нормальному |
| `numDeadRelations` -> `boolDeadRelations` | Бинаризация (>0 -> 1, =0 -> 0) | Упрощение: важен сам факт наличия мёртвых родственников, а не их точное число |
| `age` -> `age_value` + `age_no_data` | Разделение на значение и индикатор пропуска | 82% пропусков; сам факт отсутствия возраста информативен (такие персонажи чаще живы) |
| `dateOfBirth` -> удалён | Коллинеарен с `age_no_data` | У одних и тех же персонажей не указаны оба; дублирование информации |
| `isAliveSpouse` -> `spouse_alive_value` + `spouse_alive_known` | Аналогично age | 87% пропусков; разделяем значение и факт наличия данных |
| `culture` -> группировка -> one-hot | 51 уникальное значение -> 11 групп -> 12 колонок (включая `culture_no_data`) | Снижаем кардинальность через словарь регионов; one-hot для работы с моделями |
| Исправление тестовых данных | Строки 1685, 1869: аномально отрицательные `age` и `dateOfBirth` | Ошибки в данных, указанные в задании |

### Финальный набор признаков (27 штук)
- **Бинарные**: `male`, `book1`-`book5`, `isMarried`, `isNoble`, `boolDeadRelations`, `age_no_data`, `spouse_alive_value`, `spouse_alive_known`
- **Числовые**: `numDeadRelations`, `popularity_log`, `age_value`
- **One-hot (культура)**: `cult_Dorne`, `cult_Essos Nations`, `cult_Old Nations`, `cult_Other Nations`, `cult_culture_no_data`, `cult_the Iron Islands`, `cult_the Isles and Rivers`, `cult_the Mountain and the Vale`, `cult_the North`, `cult_the Reach`, `cult_the Rock`, `cult_the Stormlands`

### Самые информативные признаки
1. **boolDeadRelations** — наличие мёртвых родственников сильно снижает шансы выжить
2. **book4** — появление в 4-й книге повышает шансы выживания
3. **popularity_log** — популярные персонажи чаще погибают
4. **male** — мужчины выживают реже женщин
5. **age_no_data** — персонажи без указанного возраста чаще живы

## Модели классификации

### 1. Logistic Regression (Логистическая регрессия)

**Принцип работы**: вычисляет линейную комбинацию признаков и пропускает через сигмоидную функцию, получая вероятность принадлежности к классу 1. Порог обычно 0.5.

**Плюсы**:
- Быстрая, интерпретируемая (коэффициенты показывают вклад каждого признака)
- Хорошо работает при линейной разделимости классов
- Регуляризация (L1/L2) предотвращает переобучение

**Минусы**:
- Не улавливает нелинейные зависимости
- Требует масштабирования признаков

**Результат**: CV-10 Accuracy = **0.8060** (лучшие параметры: C=0.01, penalty=l2)

### 2. Random Forest (Случайный лес)

**Принцип работы**: ансамбль из множества деревьев решений. Каждое дерево обучается на случайной подвыборке данных (bootstrap) и случайном подмножестве признаков. Итоговое предсказание — голосование большинством.

**Плюсы**:
- Устойчив к переобучению (за счёт бэггинга и случайности)
- Не требует масштабирования
- Показывает важность признаков (`feature_importances_`)
- Хорошо работает «из коробки»

**Минусы**:
- Менее интерпретируем, чем одиночное дерево
- Может быть медленным при большом количестве деревьев

**Результат**: CV-10 Accuracy = **0.8137** (лучшие параметры: max_depth=7, min_samples_leaf=3, n_estimators=100)

### 3. KNN (K ближайших соседей)

**Принцип работы**: для нового объекта находит K ближайших объектов из обучающей выборки (по евклидову расстоянию) и присваивает класс большинства среди соседей.

**Плюсы**:
- Простой, интуитивно понятный
- Нет этапа обучения (ленивое обучение)
- Может улавливать сложные границы решения

**Минусы**:
- Медленный при предсказании (нужно считать расстояния до всех объектов)
- Очень чувствителен к масштабу признаков (обязателен `StandardScaler`)
- Плохо работает при большом числе признаков (проклятие размерности)

**Результат**: CV-10 Accuracy = **0.8105** (лучшие параметры: n_neighbors=31)

### 4. Decision Tree (Дерево решений)

**Принцип работы**: строит дерево из последовательных вопросов вида «признак X > порога?», разделяя данные на всё более однородные группы. Листья дерева содержат предсказания.

**Плюсы**:
- Максимально интерпретируем (можно визуализировать дерево)
- Не требует масштабирования
- Быстрый

**Минусы**:
- Склонен к переобучению (нужно ограничивать глубину `max_depth`)
- Нестабилен: небольшие изменения в данных меняют структуру дерева
- Обычно уступает ансамблям

**Результат**: CV-10 Accuracy = **0.8079** (лучшие параметры: max_depth=5)

## Уроки и выводы

1. **Data leakage** — признак `dateOfDeath` почти полностью дублирует target. Всегда проверяйте, нет ли признаков, которые «подсматривают» ответ.

2. **Пропуски != мусор** — сам факт отсутствия данных может быть информативным. Признак `age_no_data` оказался полезным: персонажи без указанного возраста чаще живы.

3. **Feature engineering важнее выбора модели** — разница между моделями ~1%, а правильная обработка признаков (лог-трансформация, бинаризация, группировка культур) критична для достижения хорошего результата.

4. **Группировка категорий** — при большом количестве уникальных значений группировка снижает размерность и уменьшает шум. 51 культура -> 11 групп.

5. **Масштабирование** — обязательно для моделей, основанных на расстоянии (KNN, LogReg). Деревья и ансамбли деревьев масштабирования не требуют.

6. **GridSearchCV + кросс-валидация** — перебор гиперпараметров с 10-fold CV даёт более надёжную оценку, чем подбор на одном validation split.

7. **Pipeline предотвращает ошибки** — объединение scaler + модель гарантирует, что масштабирование не «протечёт» на валидационную выборку.

8. **Одинаковый preprocessing для train и test** — оформление предобработки как функции `preprocess()` гарантирует идентичную обработку обоих датасетов.

9. **Воспроизводимость** — `random_state=42` во всех случайных операциях.

## Полезные функции

### Загрузка данных с Google Drive

```python
import gdown
gdown.download(id='FILE_ID', output='data/filename.csv', quiet=False)
```

### Анализ пропущенных значений
```python
missing = df.isnull().sum()
missing_pct = (missing / len(df) * 100).round(1)
pd.DataFrame({'Пропуски': missing, '%': missing_pct}).query('Пропуски > 0').sort_values('%', ascending=False)
```

### Логарифмическое преобразование скошенного признака
```python
# M — масштабирующий коэффициент, +1 чтобы log(0) не было проблемой
df['feature_log'] = np.log10(df['feature'] * 100 + 1)
```

### Обработка пропусков: значение + индикатор
```python
df['age_value'] = df['age'].fillna(0)
df['age_no_data'] = df['age'].isna().astype(int)
```

### Инвертирование словаря для маппинга
```python
# {группа: [значение1, значение2]} -> {значение1: группа, значение2: группа}
inverted = {}
for group, values in original_dict.items():
    for v in values:
        inverted[v] = group

df['culture_grouped'] = df['culture'].str.lower().map(inverted).fillna('no_data')
```

### One-Hot Encoding
```python
dummies = pd.get_dummies(df['category_col'], prefix='cat', dtype=int)
df = pd.concat([df.drop(columns=['category_col']), dummies], axis=1)
```

### Pipeline + GridSearchCV
```python
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.model_selection import GridSearchCV

pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('model', LogisticRegression(random_state=42, max_iter=1000, solver='liblinear'))
])

# Параметры модели в pipeline указываются через двойное подчёркивание: шаг__параметр
param_grid = {'model__C': [0.01, 0.1, 1.0], 'model__penalty': ['l1', 'l2']}

grid = GridSearchCV(pipe, param_grid, cv=10, scoring='accuracy', n_jobs=-1)
grid.fit(X, y)

print(grid.best_params_, grid.best_score_)
best_model = grid.best_estimator_
```

### Выравнивание колонок train и test после one-hot
```python
# Добавляем отсутствующие колонки (заполняем нулями)
for col in train_features:
    if col not in test_df.columns:
        test_df[col] = 0

# Упорядочиваем и убираем лишние
test_df = test_df[train_features]
```

### Оценка модели
