# Прикладное применение: основы машинного обучения

> Этот документ дополняет теоретический конспект (SUMMARY.md) практической перспективой: где и как модели и методы из урока 4.1 применяются в реальных задачах IT, аналитики данных, финансов, поиска аномалий, а также в смежных областях.

---

## 1. Логистическая регрессия

### Где применяется

**Кредитный скоринг (финансы).** Банки используют логрегрессию как стандарт для оценки вероятности дефолта. Признаки: доход, стаж, кредитная история, количество открытых счетов. Главное преимущество — интерпретируемость: регулятор (ЦБ) требует объяснить, *почему* клиенту отказано. Каждый коэффициент — это вклад конкретного признака: «отказ из-за высокой долговой нагрузки (вес +2.3) и короткого стажа (вес +1.1)».

**Fraud detection в реальном времени (финансы, IT).** Когда карточная транзакция проходит через платёжный шлюз, модель за микросекунды оценивает вероятность мошенничества по признакам: сумма, время суток, геолокация, тип мерчанта. Линейная операция (умножение вектора на матрицу) выполняется мгновенно — при миллионах транзакций в секунду это критично.

**Churn prediction (IT, e-commerce).** Предсказание оттока клиентов: уйдёт ли пользователь в следующем месяце? Признаки — активность за 30/60/90 дней, обращения в поддержку, тарифный план. Выход — вероятность от 0 до 1, по которой retention-команда расставляет приоритеты.

**Конверсия в маркетинге.** Предсказание, купит ли посетитель сайта товар после клика на рекламу. Признаки: источник трафика, устройство, время на сайте, количество просмотренных страниц. Модель помогает оптимизировать рекламный бюджет, показывая, какие каналы и аудитории конвертируются лучше.

### Когда НЕ применять

- **Нелинейные зависимости.** Если граница между классами — не прямая линия (например, аномалии расположены в центре облака нормальных данных).
- **Сложные взаимодействия признаков.** Если результат зависит от комбинации «A высокий И B низкий» — логрегрессия этого не поймает без ручного создания произведений признаков.
- **Изображения, текст, аудио.** Сырые пиксели или токены — территория нейросетей.

### Плюсы / Минусы

| Плюсы | Минусы |
|-------|--------|
| Интерпретируемость (коэффициенты = вклад) | Не ловит нелинейные зависимости |
| Мгновенный inference | Требует масштабирования (StandardScaler) |
| Выдаёт калиброванные вероятности | Чувствительна к мультиколлинеарности |
| Регулятор принимает как объяснимую модель | Требует feature engineering для сложных задач |

## 2. Random Forest (Случайный лес)

### Где применяется

**Антифрод в банках (batch-обработка).** Ежедневный анализ транзакций на предмет подозрительных паттернов. Random Forest хорошо работает с табличными данными «из коробки»: не нужно масштабирование, сам находит нелинейные зависимости и взаимодействия признаков. Feature importance показывает, что влияет на подозрительность: время суток, расстояние между транзакциями, нетипичный мерчант.

**Предиктивное обслуживание IT-инфраструктуры.** Предсказание отказа сервера или диска по метрикам мониторинга: CPU usage, disk I/O, error rate, memory pressure. Random Forest устойчив к выбросам в метриках (пиковые нагрузки) и автоматически находит комбинации признаков, предвещающие инцидент (например, «CPU > 80% И disk I/O > 90% И error rate растёт»).

**Оценка кандидатов (HR-аналитика).** Предсказание успешности кандидата по данным резюме и интервью: опыт, навыки, результаты тестового задания, количество смен работы. Random Forest даёт feature importance — какие факторы реально влияют на успех сотрудника, а какие — шум. Важно: модель должна быть проверена на bias (пол, возраст, этничность не должны влиять на решение).

**Категоризация товаров (e-commerce).** Автоматическая классификация товаров по категориям на основе числовых признаков: цена, вес, размеры, количество фото, длина описания. Random Forest справляется с мультиклассовой задачей и не требует предобработки числовых признаков.

### Когда НЕ применять

- **Real-time inference с SLA < 1 мс.** Ансамбль из 100+ деревьев медленнее одной линейной операции.
- **Очень маленькие датасеты (< 200 строк).** Склонен к переобучению, несмотря на бэггинг.
- **Нужна полная интерпретируемость.** Feature importance показывает *что* важно, но не *как* признак влияет (в отличие от коэффициентов логрегрессии).

### Плюсы / Минусы

| Плюсы | Минусы |
|-------|--------|
| Работает «из коробки» без настройки | Медленнее логрегрессии при inference |
| Не требует масштабирования | Интерпретируемость ниже, чем у логрегрессии |
| Устойчив к выбросам и шуму | Потребляет больше памяти |
| Показывает feature importance | На маленьких данных может переобучиться |
| Ловит нелинейные зависимости | Не экстраполирует за пределы обучающих данных |

## 3. KNN (K ближайших соседей)

### Где применяется

**Поиск аномалий (кибербезопасность, финансы).** KNN хорошо подходит для обнаружения аномалий по принципу «объекты, далёкие от всех своих соседей, подозрительны». Если сетевое подключение или транзакция не имеет близких аналогов в истории — это потенциальная аномалия. Вариант: вместо классификации использовать расстояние до k-го соседа как anomaly score.

**Рекомендательные системы (e-commerce, контент).** Простейший подход к рекомендациям: найти пользователей с похожим поведением (покупки, просмотры, рейтинги) и предложить то, что покупали они. Это коллаборативная фильтрация в чистом виде. Работает без обучения — удобно для холодного старта, когда данных мало.

**Матчинг похожих клиентов (маркетинг).** Найти клиентов, похожих на лучших (VIP, высокий LTV), и предложить им такие же условия. Признаки: средний чек, частота покупок, категории товаров, канал привлечения. KNN буквально делает то, что нужно — ищет ближайших «соседей» среди существующих клиентов.

**Заполнение пропусков (аналитика данных).** KNN Imputer заменяет пропуски средним значением k ближайших соседей. Часто лучше, чем заполнение медианой, потому что учитывает контекст: пропущенный доход клиента будет заполнен доходом похожих клиентов, а не средним по всей базе.

### Когда НЕ применять

- **Большие датасеты (> 100K строк).** Prediction при наивном алгоритме (brute-force) требует вычисления расстояния до каждого объекта — O(n). KD-Tree/Ball-Tree ускоряют до O(log n), но деградируют при большой размерности (> 20 признаков). На миллионе строк в высокоразмерном пространстве — секунды, не миллисекунды.
- **Высокоразмерные данные (> 50 признаков).** Проклятие размерности: в пространстве большой размерности все точки «равноудалены», и понятие «ближайшего соседа» теряет смысл.
- **Зашумлённые данные.** KNN чувствителен к шуму — один «мусорный» сосед может изменить предсказание.

### Плюсы / Минусы

| Плюсы | Минусы |
|-------|--------|
| Интуитивно понятный | Медленный при предсказании (O(n)) |
| Нет этапа обучения | Требует масштабирования (обязательно!) |
| Находит сложные границы решения | Проклятие размерности |
| Универсален: классификация, регрессия, аномалии | Чувствителен к шуму и выбросам |
| Хорош для маленьких датасетов | Не работает с категориальными признаками напрямую |

## 4. Decision Tree (Дерево решений)

### Где применяется

**Системы принятия решений (финансы, страхование).** Дерево решений буквально формализует процесс принятия решения: «если доход > X и стаж > Y — одобрить кредит, иначе — отказать». Можно визуализировать и показать бизнесу или регулятору. Это не чёрный ящик, а прозрачная логика, которую может проверить человек без ML-бэкграунда.

**Быстрая сортировка тикетов (IT Service Management).** Автоматическая маршрутизация обращений в поддержку: по тексту, категории продукта, приоритету и истории клиента определить, в какую команду направить тикет. Дерево строит простую систему правил, которую можно встроить даже в формулу Excel.

**Правила бизнес-логики (маркетинг, e-commerce).** Сегментация клиентов для таргетированных предложений. Дерево может обнаружить: «клиенты с чеком > 5000 ₽ И покупками > 3 раз в месяц И возрастом 25-45 — самая прибыльная группа». Эти правила легко перенести в CRM-систему.

**Exploratory Data Analysis (аналитика данных).** Перед построением сложной модели дерево решений помогает понять структуру данных: какие признаки наиболее информативны, какие пороговые значения разделяют классы, есть ли простые правила, объясняющие большую часть данных.

### Когда НЕ применять

- **Когда нужна стабильность.** Небольшое изменение в данных (добавление/удаление нескольких строк) может полностью изменить структуру дерева. Для production-моделей это неприемлемо.
- **Когда нужна максимальная точность.** Одиночное дерево почти всегда проигрывает ансамблям (Random Forest, XGBoost). Используйте дерево для интерпретации, а ансамбль — для предсказания.
- **Непрерывные целевые переменные с высокой точностью.** Дерево предсказывает средние значения в листьях — это ступенчатая функция, плохо аппроксимирующая плавные зависимости.

### Плюсы / Минусы

| Плюсы | Минусы |
|-------|--------|
| Максимальная интерпретируемость | Нестабильность (дерево меняется при малых изменениях данных) |
| Визуализация = объяснение для бизнеса | Склонность к переобучению без ограничения глубины |
| Не требует масштабирования | Уступает ансамблям по качеству |
| Быстрый при обучении и предсказании | Не умеет экстраполировать |
| Работает с категориальными данными | Жадный алгоритм — не находит глобальный оптимум |

## 5. Feature Engineering — практические приёмы

### Логарифмическая трансформация

**Проблема.** Распределение признака сильно скошено: большинство значений маленькие, но есть длинный «хвост» с редкими большими значениями.

**Примеры из реальных задач:**
- **Доход клиентов (финансы, HR).** 90% людей зарабатывают 30-150K, но есть 1% с доходом > 1M. Без логарифмирования модель «растягивается» на богатых, игнорируя различия в массовом сегменте.
- **Трафик на сайт (IT, маркетинг).** Большинство страниц получают 10-100 просмотров, но есть вирусные посты с миллионами. Log-трансформация позволяет модели видеть различия между «10 и 100 просмотров», а не только «100 и 1 000 000».
- **Суммы транзакций (финансы).** Покупки в магазине: 80% — до 5000 ₽, но есть покупки на 500 000 ₽. Логарифм выравнивает масштаб.
- **Количество запросов к API (IT).** Нормальные клиенты делают 10-100 запросов/час, но боты — тысячи. Log помогает модели различать нормальных клиентов между собой, а не только «нормальный vs бот».

**Формула:** `log10(x * M + 1)`, где M — масштабирующий коэффициент, +1 — защита от log(0).

### One-Hot Encoding

**Проблема.** Модели не умеют работать с текстом. Нужно превратить категорию в числа, но нельзя просто присвоить номера (1=Москва, 2=Питер) — модель решит, что Питер «больше» Москвы.

**Примеры из реальных задач:**
- **Регион клиента (финансы, e-commerce).** 85 субъектов РФ → 85 бинарных колонок. Или сгруппировать по федеральным округам (8 колонок), если данных мало — как с культурами в задании 4.1.
- **Тип устройства (IT, маркетинг).** Desktop / Mobile / Tablet → 3 колонки. Конверсия сильно зависит от устройства.
- **Отрасль компании (HR, B2B).** Финансы / IT / Ритейл / Производство → one-hot. Важно для скоринга B2B-клиентов.

**Подводный камень: выравнивание колонок train/test.** Если в тестовых данных появляется новая категория (или исчезает существующая), количество колонок не совпадёт. В задании 4.1 это решается добавлением отсутствующих колонок с нулями и удалением лишних — обязательная практика для production.

### Обработка пропусков

**Стратегия «значение + индикатор»** из задания 4.1 — одна из лучших на практике:

| Стратегия | Когда использовать | Пример |
|-----------|-------------------|--------|
| Удалить строку | Пропусков < 5%, данных много | Пропущен email в CRM из 1M записей |
| Заполнить медианой | Пропусков < 30%, признак числовой | Пропущен возраст в 15% анкет |
| Значение + индикатор | Пропусков 30-90%, сам факт пропуска информативен | Пропущен доход (не указал = скрывает = риск?) |
| Удалить признак | Пропусков > 95% | Телефон второго контакта: 97% пустых |
| KNN Imputer | Нужно точное заполнение, данных немного | Медицинские данные: пропущен анализ, но остальные показатели есть |

**Ключевой инсайт из задания:** сам факт отсутствия данных часто несёт информацию. В задании `age_no_data` оказался информативным признаком — персонажи без указанного возраста чаще живы. В бизнесе аналогично: клиент не указал доход при заявке на кредит — это уже сигнал.

### Data Leakage (утечка данных)

**Что это.** Признак, который содержит информацию из будущего или прямо кодирует целевую переменную. Модель показывает нереальное качество на обучении, но провалится в production.

**Типичные ловушки в бизнесе:**

| Ловушка | Область | Почему это leakage |
|---------|---------|-------------------|
| Дата закрытия счёта | Финансы (churn) | Заполнена = клиент уже ушёл (как dateOfDeath в задании) |
| Сумма штрафов | Кибербезопасность | Штрафы начислены после обнаружения инцидента |
| Статус заказа «возврат» | E-commerce (fraud) | Возврат оформлен после того, как фрод обнаружен |
| Количество обращений по инциденту | IT (SLA) | Обращения — следствие инцидента, а не причина |
| Итоговая оценка сотрудника | HR (прогноз увольнения) | Низкая оценка = следствие решения об увольнении |

**Как обнаружить:**
1. Подозрительно высокий accuracy (> 0.95 на реальных данных — повод насторожиться).
2. Один признак доминирует в feature importance (> 50%).
3. Задать вопрос: «Был бы этот признак доступен *на момент* предсказания в реальной системе?»

## 6. Оценка и выбор модели

### Cross-Validation: зачем на практике

**Простой train/test split ненадёжен.** Один «неудачный» split может завысить или занизить метрику. Кросс-валидация даёт *распределение*: вместо «accuracy = 0.82» вы получаете «accuracy = 0.80 ± 0.03». Это разница между «модель хорошая» и «мы не уверены».

**5-fold vs 10-fold:**
- **5-fold:** быстрее, чуть больше разброс. Стандарт для большинства задач.
- **10-fold:** точнее, но в 2 раза дольше. Используйте при маленьких датасетах (< 5K строк), как в задании 4.1.
- **Leave-One-Out:** экстремальный случай (k = n). Для очень маленьких датасетов (< 200 строк).

### GridSearchCV: подбор гиперпараметров

**На практике:** перебор всех комбинаций параметров через GridSearchCV — стандарт, но при большом пространстве поиска переходят на `RandomizedSearchCV` (случайные комбинации) или `Optuna` / `Hyperopt` (байесовская оптимизация).

**Типичные параметры для подбора:**

| Модель | Параметры | Типичный диапазон |
|--------|-----------|-------------------|
| Logistic Regression | C, penalty | C: [0.001, 0.01, 0.1, 1, 10]; penalty: [l1, l2] |
| Random Forest | n_estimators, max_depth, min_samples_leaf | n_est: [50, 100, 200]; depth: [3, 5, 7, 10, None]; leaf: [1, 3, 5] |
| KNN | n_neighbors | [3, 5, 7, 11, 15, 21, 31] |
| Decision Tree | max_depth, min_samples_leaf | depth: [3, 5, 7, 10]; leaf: [1, 3, 5, 10] |

### Метрики: когда accuracy недостаточно

Accuracy вводит в заблуждение при дисбалансе классов. Если фрод — 0.1% транзакций, модель «всегда нет» даёт accuracy = 99.9%, но бесполезна.

| Метрика | Когда использовать | Бизнес-пример |
|---------|-------------------|---------------|
| **Accuracy** | Классы сбалансированы | Классификация товаров по категориям |
| **Precision** | Цена false positive высока | Блокировка легитимной транзакции = потеря клиента |
| **Recall** | Цена false negative высока | Пропущенный фрод = финансовые потери |
| **F1-score** | Баланс precision и recall | Churn: хотим найти «уходящих», но не перегрузить retention-команду |
| **Confusion Matrix** | Детальный разбор ошибок | Fraud: сколько легитимных транзакций заблокировано vs сколько фродов пропущено — два разных вида ущерба |
| **AUC-ROC*** | Общее сравнение моделей | Какая из трёх моделей лучше для скоринга? |
| **AUC-PR*** | Сильный дисбаланс классов | Поиск аномалий (0.01% положительных) |

\* *AUC-ROC и AUC-PR выходят за рамки урока 4.1, но необходимы при работе с несбалансированными данными на практике.*

**Для поиска аномалий:** AUC-PR информативнее AUC-ROC, потому что ROC завышает качество за счёт огромного количества true negatives.

### Pipeline: защита от ошибок

**Зачем.** Без Pipeline легко допустить ошибку: обучить StandardScaler на всех данных (включая test), сделать one-hot encoding до разбиения — и получить data leakage. Pipeline гарантирует, что предобработка обучается только на train-фолде.

**В production это критично:** scaler, обученный на исторических данных, должен быть сохранён и применён к новым данным с теми же параметрами. Забыли сохранить scaler — предсказания бессмысленны.

## 7. Сводная таблица: модель → задача

| Модель | Лучшие задачи | Ограничения | Отрасли |
|--------|--------------|-------------|---------|
| **Logistic Regression** | Скоринг, fraud (real-time), churn, конверсия | Только линейные зависимости, нужен scaler | Финансы, IT, маркетинг |
| **Random Forest** | Fraud (batch), предиктивное обслуживание, категоризация | Медленнее для inference, менее интерпретируем | Все отрасли — универсальный baseline |
| **KNN** | Аномалии, рекомендации, матчинг, заполнение пропусков | Медленный на больших данных, проклятие размерности | Кибербезопасность, e-commerce, аналитика |
| **Decision Tree** | Бизнес-правила, сортировка тикетов, EDA, сегментация | Нестабильность, переобучение, уступает ансамблям | Финансы, IT, маркетинг |

### Фреймворк выбора модели

```
1. Нужна интерпретируемость для регулятора или бизнеса?
   → Да, прозрачные правила → Decision Tree
   → Да, коэффициенты → Logistic Regression

2. Inference < 1 мс?
   → Да → Logistic Regression или Decision Tree

3. Данных меньше 1000?
   → Да → Logistic Regression (L2) или KNN

4. Нужен поиск аномалий по расстоянию?
   → Да → KNN

5. Во всех остальных случаях:
   → Начать с Logistic Regression (baseline)
   → Попробовать Random Forest
   → Сравнить на кросс-валидации
   → Выбрать с учётом: качество vs интерпретируемость vs скорость
```

## 8. Чек-лист практика

| Задача | Рекомендуемый подход |
|--------|---------------------|
| Кредитный скоринг | LogReg + Pipeline(StandardScaler), стратифицированная CV-10 |
| Fraud detection (real-time) | LogReg (скорость), порог < 0.5, AUC-PR |
| Fraud detection (batch) | Random Forest / XGBoost, LogReg как baseline |
| Churn prediction | LogReg (baseline) → Random Forest, F1 / AUC-ROC |
| Поиск аномалий в логах | KNN (anomaly score = расстояние до k-го соседа) |
| Сегментация клиентов | Decision Tree (визуальные правила для бизнеса) |
| Категоризация товаров | Random Forest (мультикласс, без масштабирования) |
| Маршрутизация тикетов | Decision Tree → набор правил для CRM |
| Рекомендации (cold start) | KNN по профилю пользователя |
| Оценка кандидатов (HR) | Random Forest + feature importance для объяснения |
| Exploratory Data Analysis | Decision Tree (визуализация + feature importance) |
| Маленький датасет (< 1K) | LogReg (L2) + CV-10 |
