# Конспект — StyleCLIP (Часть 2 задания 23.1)

> Тема: редактирование изображений по тексту через оптимизацию латента StyleGAN2.
> Базовая статья: [StyleCLIP: Text-Driven Manipulation of StyleGAN Imagery](https://arxiv.org/abs/2103.17249) (Patashnik et al., 2021).

---

## 1. Задача редактирования изображений по тексту

Дано изображение лица, сгенерированное StyleGAN2. Нужно изменить его в соответствии с
текстовым описанием — например, «человек с зелёными волосами», «улыбающийся человек»,
«с бородой» — **сохранив при этом узнаваемость человека** (его идентичность).

Сложность в том, что у нас нет пар «было / стало» для обучения. Текст и изображение —
объекты из разных модальностей. StyleCLIP решает это так:

- **StyleGAN2** умеет генерировать реалистичные лица из латентного вектора. Если
  аккуратно сдвинуть латент, изменится и картинка — причём правдоподобно.
- **CLIP** умеет измерять, насколько изображение «похоже» на текст. Это даёт
  направление: в какую сторону двигать латент.
- Соединив их, можно поставить **задачу оптимизации**: ищем латент, при котором
  сгенерированное изображение максимально близко к тексту по CLIP.

### Идея StyleCLIP (вариант latent optimization)

Берём исходный латент `w_s`, делаем его обучаемой переменной `w` и градиентным спуском
минимизируем составной лосс. Веса всех нейросетей (StyleGAN2, CLIP, ArcFace)
**заморожены** — оптимизируется только сам вектор `w`. Это и есть «редактирование в
латентном пространстве»: мы не дообучаем модели, а ищем точку в их латенте.

Картина целиком:

```
текст t  ──► CLIP text encoder ──┐
                                 ├─► CLIP loss ──┐
w  ──► StyleGAN2 ──► image G(w) ─┘               │
 │                      │                        ├─► общий loss ──► backward ──► ∇w
 │                      └─► ArcFace ──► ID loss ──┤
 └────────────────► ||w - w_s||₂ ──► L2 loss ─────┘
```

## 2. Латентные пространства StyleGAN2: Z, W, W+

StyleGAN2 генерирует изображение не напрямую из шума, а через несколько латентных
пространств. Понимание разницы между ними — ключ к тому, почему оптимизируют именно в W.

| Пространство | Что это | Размерность (FFHQ) | Свойства |
|---|---|---|---|
| **Z** | Исходный шум `z ~ N(0, I)` | 512 | Распределение фиксировано (гауссиана). «Запутанное» (entangled): один атрибут размазан по многим координатам. |
| **W** | Выход mapping-сети `w = MLP(z)` | 512 | Не обязан быть гауссовым. **Расцеплённое** (disentangled): атрибуты лучше разделены, латент интерпретируемее. |
| **W+** | Свой `w` для каждого из 18 слоёв генератора | 18 × 512 | Самое выразительное пространство. Используется для GAN-инверсии реальных фото, но легко «вылетает» из реалистичного многообразия. |

**Z → W.** Mapping-сеть — это 8-слойный MLP. Она преобразует простую гауссиану `Z` в
сложно устроенное пространство `W`, где направления соответствуют осмысленным
изменениям внешности. В коде задания: `latent_w = generator.style(latent_z)`.

**Почему оптимизируют в W, а не в Z или W+:**

- В **Z** распределение жёстко гауссово, координаты сильно запутаны — двигать вектор
  под текст трудно и неустойчиво.
- **W** расцеплено: маленький сдвиг чаще меняет один атрибут, не ломая остального.
  Один `w` (512 чисел) на всё изображение — компактно, оптимизация устойчива,
  результат остаётся в области реалистичных лиц.
- **W+** даёт больше свободы (18 разных векторов), но именно из-за свободы легче
  получить артефакты и нереалистичные лица. Для оптимизации под текст W — разумный
  компромисс между выразительностью и стабильностью.

В задании оптимизация идёт в **W**: один вектор `(1, 512)`.

## 3. CLIP: связь изображения и текста

**CLIP** (Contrastive Language-Image Pre-training, OpenAI) — модель, обученная на
сотнях миллионов пар «картинка + подпись». Её задача при обучении: для правильной пары
сделать эмбеддинги картинки и текста близкими, для неправильной — далёкими
(контрастивная функция потерь).

CLIP состоит из двух энкодеров:

- **Image encoder** — Vision Transformer (в задании `ViT-B/32`) или ResNet. Принимает
  изображение, выдаёт вектор-эмбеддинг (для ViT-B/32 — 512 чисел).
- **Text encoder** — Transformer. Принимает токенизированный текст (длина 77 токенов в
  задании), выдаёт вектор той же размерности.

Главное свойство: **оба энкодера выдают векторы в одном общем пространстве**. Поэтому
изображение и текст можно сравнивать напрямую — обычным скалярным произведением.

### Косинусное сходство и CLIP loss

Близость двух эмбеддингов измеряется **косинусным сходством** — косинусом угла между
векторами (масштаб векторов не важен, важно направление):

```
cos_sim(a, b) = ⟨a, b⟩ / (‖a‖ · ‖b‖)
```

На практике сначала L2-нормируют оба вектора (делят на их норму), и тогда косинусное
сходство — просто скалярное произведение нормированных векторов. Значения:
`+1` — одинаковое направление, `0` — ортогональны, `-1` — противоположны.

**CLIP loss** = косинусное *расстояние*, то есть `1 - cos_sim`:

```
D_CLIP(G(w), t) = 1 - cos_sim( E_img(G(w)), E_text(t) )
```

- `0` — изображение идеально соответствует тексту;
- `2` — полностью противоположно.

Минимизация CLIP loss «толкает» изображение к тексту в общем CLIP-пространстве.

**Технические детали из задания.** StyleGAN2 выдаёт картинку 1024×1024, а CLIP ждёт
224×224. В классе `CLIPLoss` это решается через `Upsample(×7)` + `AvgPool2d(kernel=32)`.
Эмбеддинг изображения считается **без** `torch.no_grad()` — градиент обязан течь через
CLIP-энкодер к латенту. Эмбеддинг текста можно не отслеживать (промпт постоянен).

## 4. ArcFace и ID loss: сохранение идентичности

Если оптимизировать только CLIP loss, латент может «уплыть» так, что под промпт
подгонится **другое лицо**. Чтобы этого не было, добавляют **ID loss** — штраф за
потерю идентичности человека.

**ArcFace** — нейросеть распознавания лиц. В задании это ResNet-50 с Squeeze-Excitation
блоками (режим `ir_se`), вход 112×112. Название ArcFace — от функции потерь (Additive
Angular Margin Loss), с которой её обучали: она раздвигает классы (людей) по углам на
гиперсфере, поэтому сеть выдаёт эмбеддинг, в котором **разные люди — далеко, одно лицо
в разных кадрах — близко**.

Важная деталь: backbone ArcFace заканчивается `l2_norm`, то есть возвращает уже
**L2-нормированные** эмбеддинги. Поэтому для косинусной близости достаточно `torch.dot`
— дополнительно нормировать не нужно.

**ID loss** — косинусное расстояние между эмбеддингом исходного и редактируемого лица:

```
L_ID(w) = 1 - ⟨ R(G(w_s)), R(G(w)) ⟩
```

где `R` — ArcFace-эмбеддинг. Чем ближе к 0, тем лучше сохранена личность. Эмбеддинг
исходного лица берётся с `.detach()` — он постоянный эталон, градиент по нему не нужен;
а эмбеддинг редактируемого изображения остаётся в графе, чтобы градиент дошёл до латента.

**Извлечение эмбеддинга лица:** изображение приводится к 256×256, из центра вырезается
область с лицом `[35:223, 32:220]`, ресайзится к 112×112 и подаётся в ArcFace. В
задании ресайз делается через `F.interpolate(mode='bilinear')` — он дифференцируем и
работает на любом устройстве (см. раздел 9).

## 5. L2-регуляризация латента

**L2 loss** — штраф за отклонение оптимизируемого латента от исходного:

```
L2 = ‖w - w_s‖₂
```

где `w_s` — зафиксированный исходный латент (в коде `latent_w`, постоянный эталон).

Зачем это нужно:

- StyleGAN2 выдаёт реалистичные лица только для латентов из «обжитой» области
  пространства W. Если `w` уйдёт слишком далеко от типичных значений, картинка пойдёт
  артефактами или перестанет быть лицом.
- L2-штраф **удерживает `w` рядом с `w_s`**, то есть в реалистичной зоне, и заставляет
  оптимизацию делать **минимальную** правку, достаточную для соответствия тексту.

На шаге 0 `w == w_s`, поэтому L2 loss = 0. По мере редактирования он закономерно
растёт — это «цена» правки: чем сильнее изменение, тем дальше латент.

## 6. Полный лосс оптимизации

StyleCLIP (вариант latent optimization) минимизирует по `w` сумму трёх слагаемых:

```
arg min_w  D_CLIP(G(w), t)  +  λ₂ · ‖w - w_s‖₂  +  λ_ID · L_ID(w)
```

| Слагаемое | Роль | За что отвечает |
|---|---|---|
| `D_CLIP(G(w), t)` | **главный** лосс | Двигает изображение к тексту. Задаёт направление редактирования. |
| `λ₂ · ‖w - w_s‖₂` | регуляризация | Удерживает латент рядом с исходным → реалистичность, минимальная правка. |
| `λ_ID · L_ID(w)` | регуляризация | Сохраняет идентичность по ArcFace → не «уплыть» в чужое лицо. |

**Роль коэффициентов** (в задании `λ₂ = 0.008`, `λ_ID = 0.001`):

- **`λ₂`**: большое значение «замораживает» картинку — правка под промпт почти не
  происходит; слишком малое — латент уходит далеко, лицо искажается. `0.008` — баланс
  между силой редактирования и сохранением структуры.
- **`λ_ID`**: большое значение жёстко держит личность, но мешает заметным изменениям
  внешности; малое — допускает дрейф в другое лицо. В задании `0.001` мало: оно почти
  не сдерживает оптимизацию, и идентичность заметно дрейфует (ID loss растёт до ~0.6).
  Чтобы удержать личность строже, `λ_ID` нужно увеличивать — ценой ослабления правки.

Оба коэффициента малы — CLIP loss доминирует, а L2 и ID работают как **мягкая
регуляризация**.

## 7. Процесс оптимизации

Оптимизация — обычный градиентный спуск, но обучаемый параметр здесь **не веса сетей,
а сам латентный вектор**:

```python
latent = latent_w.detach().clone()      # копия исходного w
latent.requires_grad = True             # делаем латент обучаемым

optimizer = torch.optim.Adam([latent], lr=0.08)          # Adam обновляет ТОЛЬКО латент
scheduler = torch.optim.lr_scheduler.ExponentialLR(optimizer, gamma=0.99)

for step in range(num_steps):                            # 200 шагов
    optimizer.zero_grad()
    img_gen, _ = generator([latent], input_is_latent=True, randomize_noise=False)
    loss = clip_loss(img_gen, text) + λ₂ * l2_loss + λ_ID * id_loss
    loss.backward()                     # градиент доходит до latent
    optimizer.step()                    # шаг по латенту
    scheduler.step()                    # снижаем lr
```

- **Adam по латенту.** В оптимизатор передаётся `[latent]` — список из одного тензора.
  Веса StyleGAN2/CLIP/ArcFace в него не входят и не меняются.
- **ExponentialLR.** На каждом шаге `lr ← lr · gamma`. При `gamma = 0.99` learning rate
  плавно затухает: к шагу 200 он составит `0.08 · 0.99²⁰⁰ ≈ 0.0107` — крупные правки в
  начале, тонкая подстройка в конце.
- **Дифференцируемость.** Вся цепочка `latent → StyleGAN2 → CLIP/ArcFace → loss`
  дифференцируема, поэтому `loss.backward()` доносит градиент текстового сигнала до
  латента. Если хоть где-то поставить `torch.no_grad()` на пути, граф порвётся и
  оптимизация работать не будет.
- **input_is_latent=True.** Говорит генератору, что ему подают вектор из W (а не шум
  из Z) — то есть mapping-сеть пропускается.

## 8. Результаты задания

Промпт: `a person with green hair`. Гиперпараметры: `lr=0.08`, `ExponentialLR gamma=0.99`,
`λ₂=0.008`, `λ_ID=0.001`, `num_steps=200`.

### Динамика лоссов за 200 шагов

| Лосс | Шаг 0 | Финал (шаг 199) | Изменение |
|---|---|---|---|
| **CLIP loss** | 0.7773 | 0.5430 | снизился на **30.2 %** |
| **L2 loss** | 0.0000 | 7.5017 | вырос (латент отошёл от `w_s`) |
| **ID loss** | 0.0365 | 0.5664 | вырос (идентичность дрейфует) |
| **Суммарный (all)** | 0.7774 | 0.6035 | снизился |

Суммарный loss каждые 50 шагов: `0.7774 → 0.6644 → 0.6168 → 0.6233 → финал 0.6035`.

### Что это означает

- **CLIP loss** в целом снижается `0.7773 → 0.5430` — изображение в CLIP-пространстве
  сдвинулось к тексту. Визуально на лице появляется зелёный оттенок волос. CLIP loss
  **не достигает нуля** — это нормально: косинусная близость реального лица и короткой
  текстовой фразы ограничена сверху, важна **относительная** динамика.
- **L2 loss** растёт от 0 (на шаге 0 `latent == latent_w`) до 7.5017 — это цена
  редактирования: латент отошёл от исходного, чтобы изменить картинку.
- **ID loss** заметно растёт `0.0365 → 0.5664`: косинусная близость ArcFace-эмбеддингов
  упала с ~0.96 до ~0.43 — идентичность ощутимо дрейфует. Малый `λ_ID=0.001` почти не
  сдерживает оптимизацию (вклад слагаемого `λ_ID·L_ID ≈ 0.0006` ничтожен); общую
  структуру лица удерживает прежде всего L2-регуляризация латента.
- **Суммарный loss** повторяет форму CLIP loss — λ₂ и λ_ID малы, L2/ID-слагаемые
  работают как мягкая регуляризация.
- **Форма кривых:** CLIP loss быстро падает в первые ~30–50 шагов, затем выходит на
  плато (ExponentialLR гасит lr).

## 9. Три варианта StyleCLIP из статьи

В оригинальной работе предложено три способа редактирования. В задании реализован
только первый.

| Вариант | Как работает | Скорость | Когда применять |
|---|---|---|---|
| **Latent optimization** (реализован) | Для каждой картинки и промпта — отдельный градиентный спуск по латенту (200 шагов). | Медленно: оптимизация заново на каждый запрос. | Произвольный текст, разовое редактирование. |
| **Latent mapper** | Обучается отдельная сеть-маппер, которая по латенту сразу выдаёт нужный сдвиг под фиксированный промпт. | Быстро на инференсе (один проход), но маппер надо обучить. | Один и тот же тип правки для многих изображений. |
| **Global directions** | Заранее находится одно универсальное направление в латентном пространстве для текстового атрибута. Правка = сдвиг вдоль него. | Очень быстро, без оптимизации на инференсе. | Атрибут известен заранее, нужна интерактивность и контроль силы правки. |

## 10. Уроки и выводы

- **StyleCLIP соединяет три замороженные сети.** StyleGAN2 генерирует, CLIP задаёт
  направление по тексту, ArcFace охраняет идентичность. Обучаемый объект — только
  латентный вектор.
- **Оптимизация в W**, а не в Z или W+: W расцеплено (атрибуты разделены) и устойчиво,
  один вектор 512 чисел держит изображение в реалистичной зоне.
- **CLIP loss = 1 − косинусное сходство** эмбеддингов картинки и текста; важна
  относительная динамика, а не достижение нуля.
- **Регуляризаторы обязательны.** Без L2 латент уходит в нереалистичную зону, без ID
  loss редактирование может подменить лицо. Коэффициенты λ₂, λ_ID — баланс между силой
  правки и сохранением исходного.
- **Сквозная дифференцируемость** — суть метода. Любой `no_grad` на пути
  `latent → … → loss` ломает оптимизацию.
- **Метод per-image и медленный** — для каждого изображения/промпта оптимизация заново.
  Для ускорения в статье есть latent mapper и global directions.

## 11. Полезные приёмы (адаптация под локальный запуск)

- **Патч CUDA-расширений StyleGAN2 под CPU/MPS.** Репозиторий `rosinality/stylegan2-pytorch`
  использует кастомные CUDA-операции (`fused_act`, `upfirdn2d`), которые не
  компилируются на macOS. В решении файлы `op/*.py` патчатся идемпотентно: вызов
  `load(...)` заменяется на `None`, а условие выбора native-ветки расширяется на
  устройства `cpu` и `mps`. StyleGAN2 работает на чистом PyTorch без CUDA-расширений.
- **`F.interpolate` вместо `AdaptiveAvgPool2d` на MPS.** Adaptive-пулинг на MPS требует
  делимости размеров (вырезанное лицо 188×188 не делится на 112). Билинейный ресайз
  `F.interpolate(x, size=(112,112), mode='bilinear')` работает на любом устройстве и
  **дифференцируем** — градиент через него проходит.
- **Заморозка весов с сохранением градиента ко входу.** `for p in model.parameters():
  p.requires_grad_(False)` отключает обновление *весов*, но НЕ блокирует граф: градиент
  всё равно течёт **через** слой к его входу (к латенту). Это не то же, что
  `torch.no_grad()` — последний рвёт граф полностью и для StyleCLIP недопустим на пути
  оптимизации.
- **Эталон с `.detach()`.** Исходный латент `w_s` и исходные ArcFace-эмбеддинги берутся
  через `.detach()` — они постоянны, градиент по ним не нужен, отвязка от графа
  экономит память.
- **Воспроизводимость.** Фиксированные сиды (`torch.manual_seed`) — иначе исходное
  «усреднённое» лицо будет каждый раз разным.
- **device-цепочка** `cuda → mps → cpu` и `map_location=device` при `torch.load` —
  ноутбук переносим между машинами без правок.
