# Прикладное применение: генеративно-состязательные сети (GAN)

> Этот документ дополняет теоретический конспект (SUMMARY.md) практической перспективой: где и как методы из урока 23.1 — GAN, DCGAN, генератор как декодер латента, дискриминатор как детектор «реальное/фейк», leave-one-out 1-NN метрика и t-SNE — применяются в реальных задачах IT/DevOps, аналитики данных, финансов, поиска аномалий, компьютерного зрения и медиа.

---

## 1. GAN (генеративно-состязательные сети) в целом

### Где применяется

**Синтетические датасеты для обучения моделей (аналитика данных, ML-инжиниринг).** Самый частый промышленный сценарий — генерация «искусственных» данных там, где реальных мало или их нельзя использовать. Банк хочет обучить модель fraud detection, но мошеннических транзакций — 0.1% датасета, и каждый банк боится делиться ими с подрядчиком. GAN, обученный на реальных транзакциях, генерирует синтетический набор той же статистической структуры, который можно отдать вендору или выложить в открытый бенчмарк. Аналогично в медицине: больница не имеет права передавать снимки пациентов, но синтетические КТ-снимки, сгенерированные GAN, формально не относятся ни к одному реальному человеку и обходят ограничения GDPR/HIPAA. Это называется data anonymization через генерацию.

**Аугментация данных в компьютерном зрении.** Когда классов много, а примеров на редкий класс — единицы (классическая проблема long-tail распределений: дефектные детали на конвейере, редкие болезни на медицинских снимках, экзотические дорожные ситуации для автопилота), GAN дописывает недостающие примеры. В отличие от простых аугментаций (поворот, отражение, ColorJitter), GAN порождает по-настоящему новые объекты, а не геометрические вариации одного и того же. Это поднимает recall на редких классах, который иначе проседает из-за дисбаланса.

**Заполнение пропусков и rebalancing в табличных данных (финансы, аналитика).** Табличные GAN (CTGAN, TVAE-семейство) генерируют строки таблиц со смешанными типами (числа + категории), сохраняя корреляции между колонками. Применяется для балансировки выборок перед обучением скоринговых моделей и для «дозаполнения» small-data сегментов (новый продукт, новый регион, где исторических данных ещё нет).

### Где НЕ стоит применять

- **Когда нужна оценка плотности или правдоподобия.** GAN — это семплер, он умеет порождать объекты, но НЕ выдаёт `p(x)` — вероятность конкретного объекта. Если задача — детекция аномалий через «насколько объект вероятен» или оценка likelihood для статистического вывода, нужны VAE, normalizing flows или autoregressive-модели, а не GAN.
- **Маленький датасет (сотни примеров).** GAN требует тысяч-десятков тысяч примеров, чтобы выучить распределение. На 200 объектах он либо переобучится и начнёт буквально копировать обучающую выборку (тогда «синтетические» данные не анонимны и бесполезны), либо схлопнется в mode collapse.
- **Задачи, где критична верифицируемость.** Синтетические медицинские или финансовые данные могут содержать незаметные артефакты, которых не бывает в реальности. Модель, обученная на них, выучит эти артефакты. Для регулируемых отраслей синтетика — вспомогательный, а не основной источник данных.
- **Когда у задачи есть простое решение.** Если данных достаточно и они доступны — GAN не нужен. Это тяжёлый, нестабильный в обучении инструмент; разворачивать его ради того, что решается обычными аугментациями или сбором данных, — overengineering.

### Типичные ошибки на практике

1. **Игнорировать mode collapse.** Генератор находит несколько «удобных» объектов, обманывающих дискриминатор, и выдаёт почти только их. Визуально картинки выглядят нормально, но разнообразие потеряно. На синтетическом датасете это значит, что вместо 42 классов персонажей вы получили 5 — и модель, обученная на таком датасете, провалится. Проверять разнообразие нужно явно: метрикой (recall/coverage), t-SNE, кластеризацией сгенерированных объектов.
2. **Доверять «красивым» картинкам без количественной метрики.** Человеческий глаз — плохой и предвзятый оценщик. Изображение может выглядеть реалистично, но статистически распределение сгенерированных объектов смещено. Всегда нужна численная метрика (FID, leave-one-out 1-NN, precision-recall).
3. **Утечка реальных данных в «синтетику».** Если GAN переобучился, он генерирует почти точные копии обучающих примеров. Для anonymization это провал безопасности: «синтетический» датасет содержит реальных пациентов/клиентов. Перед публикацией синтетики обязательно проверяют, нет ли среди сгенерированных объектов near-duplicate реальных (например, тем же 1-NN по расстоянию).
4. **Обучать GAN «как обычную сеть» и ждать падения лосса к нулю.** Лосс GAN не сходится монотонно — он колеблется (см. SUMMARY.md, раздел 1). Инженер, привыкший к supervised-задачам, видит «несходящийся» лосс, паникует и крутит learning rate, разрушая хрупкое равновесие двух сетей.

## 2. DCGAN (свёрточная архитектура генератора/дискриминатора)

### Где применяется

**Baseline-генератор изображений в продуктовых командах.** DCGAN — это «honest baseline» для генерации картинок: набор архитектурных рецептов (`ConvTranspose2d`/`Conv2d`-stride вместо pooling, BatchNorm, ReLU в генераторе / LeakyReLU в дискриминаторе, инициализация `N(0, 0.02)`), которые позволяют GAN на изображениях завестись стабильно. Когда медиа- или e-commerce-команде нужно быстро проверить гипотезу «можно ли сгенерировать товарные изображения / аватары / текстуры», начинают с DCGAN: он простой, обучается на одной GPU, и если он не справляется — есть смысл переходить к StyleGAN или diffusion. Если DCGAN дал приемлемое качество, дорогую архитектуру можно не разворачивать.

**Генерация текстур и ассетов для геймдева и 3D (медиа, IT).** DCGAN-подобные сети генерируют тайлящиеся текстуры (камень, трава, металл), вариации спрайтов, фоновые элементы. Разрешение 64-128 px, на котором DCGAN работает уверенно, как раз подходит для текстурных тайлов. Художник получает не один ассет, а семейство вариаций из шумового вектора.

**Учебная и исследовательская база для domain-specific GAN.** DCGAN-скелет (свёрточный генератор-декодер + свёрточный дискриминатор) лежит в основе множества прикладных модификаций: pix2pix для image-to-image translation (раскраска, преобразование скетча в фото), SRGAN для super-resolution (восстановление детализации на спутниковых и медицинских снимках, апскейл видео), CycleGAN для переноса стиля без парных данных. Понимание DCGAN — необходимый фундамент для работы с любой из них.

### Где НЕ стоит применять

- **Высокое разрешение (512×512 и выше).** Чистый DCGAN на больших разрешениях нестабилен и даёт артефакты (характерная «шахматная» текстура от `ConvTranspose2d`). Для фотореалистичных лиц в высоком разрешении нужны StyleGAN2/3 с прогрессивным ростом или diffusion-модели.
- **Условная генерация со сложным контролем.** Если нужно генерировать «лицо такого-то возраста, с такой эмоцией, в таком ракурсе» — базовый DCGAN не даёт рычагов управления. Нужны conditional-варианты (cGAN, AC-GAN) или модели с управляемым латентным пространством.
- **Данные не-сеточной природы.** DCGAN построен на 2D-свёртках и предполагает, что данные — это растровая сетка пикселей. Для табличных данных, графов, последовательностей свёрточная архитектура не подходит — там свои разновидности GAN.

### Типичные ошибки на практике

1. **Рассогласование диапазона генератора и данных.** Самая частая ошибка новичка: `Tanh` на выходе генератора даёт `[-1, 1]`, а реальные изображения подают в `[0, 1]` (только `ToTensor()`, без `Normalize([0.5], [0.5])`). Дискриминатор тогда отличает фейк от реального тривиально — по диапазону значений, — и обучение бессмысленно. Диапазоны генератора и данных обязаны совпадать.
2. **BatchNorm не на тех слоях.** DCGAN-рецепт: BatchNorm НЕ ставится на выход генератора и на вход дискриминатора. Если поставить — обучение дестабилизируется. Также BatchNorm плохо работает на маленьких батчах (нужен `drop_last=True` и batch_size хотя бы 32-64).
3. **Дисбаланс силы сетей.** Если дискриминатор обучается заметно быстрее генератора, он становится «идеальным», `D(G(z)) → 0`, и градиент генератора затухает. Симптом — лосс генератора неограниченно растёт. Лечится понижением learning rate дискриминатора, label smoothing, либо non-saturating loss (генератору подставляют метку 1 для фейков — как сделано в решении).
4. **Забыть `.detach()` при шаге дискриминатора.** Без `fake.detach()` градиент шага `D` потечёт в генератор и испортит его веса. Дискриминатор и генератор должны обновляться раздельными оптимизаторами и раздельными графами.
5. **Симметричная инициализация / дефолтная инициализация PyTorch.** DCGAN требует именно `N(0, 0.02)` для свёрток. На дефолтной инициализации сеть заводится менее стабильно — это эмпирически подобранный режим.

## 3. Генератор как декодер из латентного пространства (генерация синтетических данных)

### Где применяется

**Латентная арифметика и редактирование атрибутов (медиа, креативные продукты).** Генератор — это отображение `латентный вектор z → объект`. Если латентное пространство обучено хорошо, в нём работает векторная арифметика: можно найти направление «улыбка», «возраст», «очки» и двигаться вдоль него, редактируя сгенерированное лицо. На этом построены продуктовые фичи фоторедакторов (состарить/омолодить лицо, добавить улыбку), генерация аватаров с настраиваемыми атрибутами, инструменты для концепт-художников. Интерполяция между двумя точками латента даёт плавный morphing — переход от одного объекта к другому.

**Контролируемая генерация данных под нужный сценарий (тестирование, симуляция).** Поскольку выход управляется входным вектором, можно целенаправленно генерировать данные для покрытия редких случаев. Команда автопилота через генератор «дозаказывает» сцены с редкими погодными условиями; QA-команда генерирует синтетические документы/лица для нагрузочного и интеграционного тестирования пайплайна распознавания, не используя данные реальных людей. Латентный вектор здесь — это «ручка», которой задают параметры генерируемого объекта.

**Сжатие и представление данных (аналитика).** Латентный вектор — это компактное представление объекта (в решении лицо 3×128×128 = 49152 значения порождается из вектора 128). Хотя у чистого GAN нет энкодера (обратного отображения «объект → латент»), в связках GAN с энкодером (BiGAN, encoder-decoder схемы) латент используется как признаковое пространство для кластеризации, поиска похожих объектов, детекции аномалий.

### Где НЕ стоит применять

- **Когда нужно закодировать конкретный существующий объект.** У чистого GAN-генератора нет энкодера: он умеет идти `z → объект`, но не `объект → z`. Чтобы найти латент данной фотографии, нужна отдельная процедура (GAN inversion — оптимизация по `z`, либо отдельно обученный энкодер). Если задача в первую очередь «закодировать вход», берите автоэнкодер/VAE — у них энкодер встроен.
- **Когда нужна гарантированная структура латента.** У GAN латентное пространство не регуляризовано — нет гарантии, что близкие `z` дают близкие объекты или что пространство заполнено равномерно. У VAE латент регуляризуется KL-дивергенцией к `N(0, 1)`, и для задач, где важна предсказуемая геометрия латента (плавная интерполяция, семплирование), VAE надёжнее.
- **Интерпретируемость направлений «из коробки».** Латент DCGAN не размечен: какое направление отвечает за «улыбку», заранее неизвестно — его нужно искать (по размеченным примерам или unsupervised-методами вроде GANSpace). Если нужны явные, заранее заданные оси управления — это conditional-генерация, а не обычный GAN.

### Типичные ошибки на практике

1. **Семплировать из неправильного распределения.** Генератор обучен на `z ~ N(0, 1)`. Если на инференсе подать вектор из другого распределения (равномерного, с другим масштабом) — качество резко падает: сеть никогда не видела таких входов. Распределение шума на обучении и инференсе обязано совпадать.
2. **Ожидать осмысленности от линейной интерполяции в «дырах» латента.** Если двигаться между двумя `z` через область, которую генератор при обучении почти не посещал, по пути появятся артефактные объекты. Для интерполяции на сфере иногда используют slerp вместо линейной — это держит норму вектора постоянной.
3. **Путать «генератор не переобучился» с «латент содержит весь датасет».** Хороший генератор покрывает распределение, а не запоминает примеры. При mode collapse большой кусок латентного пространства схлопывается в одну-две картинки — формально латент «есть», но разнообразия нет.
4. **Считать, что генератор воспроизводит редкие моды.** Генератор тяготеет к частым, «лёгким» модам распределения. Редкие подтипы объектов (нестандартные ракурсы, экзотические классы) он порождает плохо или не порождает вовсе — для синтетического датасета это значит провал именно на хвосте, который чаще всего и нужен.

## 4. Дискриминатор как детектор «реальное/синтетическое»

### Где применяется

**Детекция дипфейков и сгенерированного контента (медиа, информационная безопасность, доверие и безопасность платформ).** Дискриминатор по своей природе — бинарный классификатор «настоящее/сгенерированное». Эта же архитектура (свёрточная сеть, выдающая вероятность подлинности) лежит в основе детекторов дипфейков: соцсети и медиаплатформы прогоняют загружаемые фото и видео через классификатор, обученный отличать реальные изображения от продуктов GAN/diffusion. Антифрод в KYC-процессах (проверка личности при открытии счёта) использует похожий детектор, чтобы отсеять поддельные/сгенерированные селфи и фото документов.

**Anomaly detection через «реконструкцию vs реальность» (поиск аномалий, промышленность, IT).** Идея дискриминатора — «насколько объект похож на распределение нормальных данных» — напрямую переносится в детекцию аномалий. GAN (или AnoGAN-подобная схема) обучается только на нормальных данных; дискриминатор и ошибка реконструкции генератора вместе дают anomaly score. Применяется в промышленном контроле качества (бракованная деталь на конвейере не похожа на распределение годных), в мониторинге IT-инфраструктуры (аномальный паттерн метрик), в медицинской диагностике (патология на снимке отклоняется от нормы). Объект, который дискриминатор уверенно метит как «не из распределения», — кандидат в аномалии.

**Perceptual loss и контроль качества генерации (компьютерное зрение).** Обученный дискриминатор используется как «компонент функции потерь» для других задач: в super-resolution и image-to-image translation adversarial loss от дискриминатора заставляет результат выглядеть реалистично, а не просто минимизировать пиксельную ошибку (которая даёт размытые картинки). Дискриминатор здесь — обучаемая метрика «похоже на настоящее».

### Где НЕ стоит применять

- **Детекция контента от модели другого типа/поколения.** Дискриминатор учится ловить артефакты конкретного генератора. Детектор, обученный против DCGAN-фейков, плохо ловит изображения от diffusion-моделей или новой версии StyleGAN — артефакты другие. Детекторы сгенерированного контента быстро устаревают и требуют постоянного дообучения; это вечная «гонка вооружений».
- **Когда нужна калиброванная вероятность.** Выход дискриминатора (после Sigmoid) формально лежит в `[0, 1]`, но это НЕ калиброванная вероятность «это фейк с такой-то вероятностью». В состязательном обучении дискриминатор смещён. Для принятия решений по порогу его выход нужно откалибровать на отложенной выборке.
- **Аномалии «внутри» нормального распределения.** Дискриминатор/GAN-anomaly-detection ловит объекты, лежащие далеко от нормы. Аномалию, которая по всем признакам выглядит нормальной, но аномальна по смыслу (мошенник, идеально имитирующий поведение обычного клиента), такой подход не поймает.

### Типичные ошибки на практике

1. **Использовать дискриминатор после обучения без переобучения под задачу.** Дискриминатор из обученного GAN заточен под конкретного «своего» генератора. Как универсальный детектор фейков он работает плохо — его нужно дообучать на актуальном наборе реальных и сгенерированных примеров.
2. **Игнорировать сдвиг домена.** Детектор дипфейков, обученный на лицах знаменитостей в студийном освещении, провалится на селфи с телефона в плохом свете. Распределение «реальных» данных в обучении детектора должно совпадать с продакшеном.
3. **Один порог 0.5 для всех сценариев.** Цена ошибок несимметрична: в KYC ложноположительное срабатывание (заблокировать настоящего клиента) и ложноотрицательное (пропустить поддельное селфи) стоят по-разному. Порог настраивают под бизнес-цену ошибок (как в задаче fraud detection — см. USAGE.md задания 6.1).
4. **Доверять дискриминатору как единственному сигналу.** В anomaly detection дискриминатор шумный. Надёжнее комбинировать его с ошибкой реконструкции генератора и другими сигналами, а не полагаться на один скаляр.

## 5. Leave-one-out 1-NN classifier accuracy как метрика качества генеративной модели

### Где применяется

**Приёмочная проверка синтетических датасетов (аналитика данных, ML-инжиниринг).** Leave-one-out 1-NN accuracy — это «two-sample test»: метрика отвечает на вопрос «отличимо ли распределение сгенерированных объектов от реального». Берут `N` реальных и `N` синтетических объектов, метят 1 и 0, и по схеме leave-one-out обучают 1-NN классификатор. Идеал — accuracy около **0.5** (объекты неотличимы, ближайший сосед случайно реальный или фейковый); accuracy около **1.0** — синтетика легко отделяется, генератор плохой. Это удобный gate в пайплайне: прежде чем отдать синтетический датасет на обучение модели, прогоняют 1-NN тест — если accuracy высокая, датасет статистически не похож на реальный, и обучать на нём нет смысла.

**Детекция переобучения GAN и утечки данных (безопасность данных).** У 1-NN метрики есть второй, важный для безопасности смысл. Accuracy сильно НИЖЕ 0.5 — подозрительный сигнал: он означает, что сгенерированные объекты подозрительно близки к реальным, вплоть до того, что ближайшим соседом реального объекта оказывается его сгенерированная почти-копия. Для anonymization это красный флаг — GAN запомнил обучающую выборку, и «синтетика» содержит реальных людей. Раздельный анализ 1-NN accuracy на реальной и на синтетической половине (LOO-test для каждого класса отдельно) показывает, переобучился ли генератор.

**Сравнение генеративных моделей на не-картиночных данных (финансы, табличные данные).** В отличие от FID/Inception Score, которые завязаны на сети, предобученные для изображений, 1-NN тест работает на любых векторных представлениях — табличных строках, эмбеддингах, временных рядах. Поэтому для оценки табличных GAN (синтетические транзакции, синтетические записи клиентов) leave-one-out 1-NN — естественный выбор: посчитал расстояния, прогнал классификатор, получил число.

### Где НЕ стоит применять

- **Большие выборки без подвыборки.** Полный leave-one-out — это `O(N²)` по расстояниям. На десятках тысяч объектов это неприемлемо долго (в решении именно поэтому взята подвыборка 600+600 и уменьшение картинок до 32×32). Для больших `N` либо берут подвыборку, либо переходят к метрикам, которые так не масштабируются.
- **Высокоразмерное сырое пространство.** 1-NN опирается на расстояния, а в высокой размерности расстояния «схлопываются» (проклятие размерности — все объекты примерно равноудалены). На сырых пикселях 128×128×3 метрика шумная; в решении изображения сжимают до 32×32. Надёжнее считать 1-NN не на сырых данных, а на осмысленных признаках/эмбеддингах.
- **Когда нужна оценка перцептивного качества и детализации.** 1-NN тест отвечает на «отличимы ли распределения», но не на «выглядит ли картинка чётко и реалистично для человека». Для перцептивной оценки изображений FID информативнее.
- **Сильно несбалансированные выборки.** Метрика предполагает равное число реальных и фейковых. При дисбалансе accuracy становится вводящей в заблуждение (как обычная accuracy на несбалансированных классах — см. USAGE.md задания 6.1).

### Типичные ошибки на практике

1. **Считать «чем ниже accuracy, тем лучше».** Это главная ловушка. Идеал — 0.5, а НЕ 0. Accuracy около 0 или сильно ниже 0.5 означает не «отличный генератор», а переобучение/копирование данных. Целиться надо в 0.5 с обеих сторон.
2. **Считать метрику на сырых высокоразмерных данных.** Без снижения размерности (resize, PCA, эмбеддинги) 1-NN в тысячемерном пространстве шумит. Шаг сжатия признаков — обязательный, а не косметический.
3. **Не фиксировать подвыборку.** Если каждый запуск метрики берёт случайные 600 объектов без `random_state`, accuracy «гуляет», и сравнивать две модели невозможно. Подвыборку и сиды надо фиксировать (в проекте — `random_state=42`).
4. **Использовать как единственную метрику.** 1-NN accuracy не ловит mode collapse: генератор может схлопнуться в несколько объектов, и если эти объекты статистически близки к реальным, accuracy будет около 0.5 — «хорошо», хотя разнообразие потеряно. Нужна пара «качество + разнообразие» (например, precision-recall для генеративных моделей).

## 6. t-SNE для визуальной оценки совпадения распределений

### Где применяется

**Качественная диагностика генеративной модели (ML-инжиниринг, аналитика).** t-SNE снижает размерность до 2D, сохраняя локальную структуру (близкие объекты остаются близкими). Если спроецировать реальные и сгенерированные объекты в одно 2D-пространство и раскрасить по происхождению, картина читается мгновенно: точки перемешаны — распределения близки; два обособленных облака — генератор систематически отличается от реальных данных. Это наглядная иллюстрация к численной метрике из раздела 5: сильное перекрытие облаков соответствует 1-NN accuracy около 0.5. Такой scatter — стандартный слайд в отчёте о качестве генеративной модели, понятный и не-инженеру.

**Диагностика mode collapse и покрытия мод (отладка GAN).** t-SNE показывает то, чего не видит 1-NN accuracy. Если реальные данные на проекции образуют несколько кластеров (разные типы объектов), а сгенерированные сбились в один-два — это визуальный диагноз mode collapse: генератор покрыл не все моды распределения. Инженер видит на одном графике, какие именно области реального распределения генератор «пропустил».

**Общая разведка эмбеддингов и кластеров (аналитика данных, поиск аномалий).** Вне GAN t-SNE — рабочая лошадка для визуализации любых высокоразмерных представлений: эмбеддинги пользователей в рекомендательных системах, представления документов в NLP, латентные коды автоэнкодеров. Аномалии часто видны как точки, оторванные от основных кластеров; новые/редкие сегменты пользователей — как обособленные группы. Это инструмент быстрой визуальной гипотезы перед формальным анализом.

### Где НЕ стоит применять

- **Как количественную метрику.** t-SNE — это визуализация, а не число. «Облака перекрываются» нельзя положить в SLA, дашборд или автоматический gate. Для численного решения нужны 1-NN accuracy, FID, precision-recall. t-SNE их дополняет, но не заменяет.
- **Интерпретация глобальной геометрии — расстояний между кластерами и их размеров.** t-SNE достоверно передаёт только локальную структуру. Расстояние между двумя кластерами на t-SNE-картинке, их относительные размеры и плотность НЕ имеют буквального смысла — алгоритм их искажает. Делать вывод «класс A в три раза дальше от B, чем от C» по t-SNE нельзя.
- **Воспроизводимые пайплайны без фиксации параметров.** Результат t-SNE зависит от `perplexity`, `random_state`, инициализации, числа итераций. Без фиксации этих параметров два запуска дадут разные картинки, и сравнивать их некорректно.
- **Очень большие выборки в реальном времени.** t-SNE вычислительно тяжёл и плохо масштабируется на сотни тысяч точек. Для интерактивной визуализации больших данных чаще берут UMAP (быстрее, лучше сохраняет глобальную структуру).

### Типичные ошибки на практике

1. **Читать расстояния и размеры кластеров буквально.** Самая частая и опасная ошибка. «Эти два облака далеко → распределения сильно разные» — некорректный вывод: t-SNE искажает глобальные расстояния. Достоверно только «перемешаны / не перемешаны» локально.
2. **Не подбирать `perplexity`.** Параметр задаёт баланс локальной и глобальной структуры (типично 5-50). Слишком маленький — картинка рассыпается на мелкие комки, слишком большой — всё сливается. На разных `perplexity` стоит посмотреть несколько раз.
3. **Сравнивать t-SNE-картинки, посчитанные раздельно.** t-SNE не имеет фиксированной системы координат. Чтобы сравнивать реальные и сгенерированные объекты, их проецируют ОДНИМ запуском `fit_transform` на объединённой выборке (как сделано в решении), а не двумя независимыми.
4. **Считать t-SNE на сырых высокоразмерных данных без предобработки.** На сырых пикселях t-SNE шумит и работает медленно. Обычно сначала снижают размерность (PCA, эмбеддинги, resize), потом t-SNE — это и быстрее, и чище.
5. **Делать выводы по одной случайной инициализации.** Один прогон t-SNE может дать нерепрезентативную раскладку. Перед выводами стоит прогнать несколько раз с разными сидами и убедиться, что картина устойчива.

## 7. Сравнение с альтернативами

### Выбор генеративной модели: GAN vs VAE vs diffusion vs autoregressive

| Критерий | GAN | VAE | Diffusion | Autoregressive |
|---|:---:|:---:|:---:|:---:|
| **Качество/резкость изображений** | Высокое, резкие | Размытые | Очень высокое (SOTA) | Высокое |
| **Разнообразие, покрытие мод** | Слабое (mode collapse) | Хорошее | Отличное | Отличное |
| **Стабильность обучения** | Низкая (хрупкое равновесие) | Высокая | Высокая | Высокая |
| **Скорость генерации (инференс)** | Очень быстрая (1 проход) | Очень быстрая | Медленная (десятки шагов) | Медленная (по элементу) |
| **Оценка правдоподобия `p(x)`** | Нет | Приближённо (ELBO) | Приближённо | Точно |
| **Энкодер `объект → латент`** | Нет (нужен GAN inversion) | Есть встроенный | Нет | Нет |
| **Структурированный латент** | Нет | Да (регуляризован KL) | — | — |

**Когда что выбирать:**

- **GAN** — когда нужны резкие, реалистичные сэмплы и быстрая генерация, а оценка плотности и встроенный энкодер не важны: генерация лиц/текстур/ассетов, аугментация изображений, super-resolution, image-to-image translation. Цена — нестабильное обучение и риск mode collapse.
- **VAE** — когда нужен структурированный латент и встроенный энкодер: кодирование объектов, плавная интерполяция, латентная арифметика с предсказуемой геометрией, anomaly detection через ошибку реконструкции. Цена — размытые изображения.
- **Diffusion-модели** — когда приоритет максимальное качество и разнообразие, а медленный инференс терпим: продуктовая фотогенерация, синтез изображений по тексту. Сегодня это SOTA по качеству картинок.
- **Autoregressive-модели** — когда данные естественно последовательны (текст, аудио, временные ряды) и нужна точная оценка правдоподобия. Для изображений медленны, но дают точный likelihood.

**Гибриды.** На практике границы размыты: VAE-GAN объединяет энкодер VAE с резкостью GAN; latent diffusion применяет diffusion в латентном пространстве автоэнкодера, ускоряя генерацию.

### Выбор метрики качества генеративной модели

| Метрика | Что измеряет | Когда выбрать | Ограничение |
|---|---|---|---|
| **Leave-one-out 1-NN accuracy** | Отличимость распределений (two-sample test); идеал ~0.5 | Табличные данные, эмбеддинги; контроль переобучения/утечки; нет картиночной предобученной сети | `O(N²)`; не ловит mode collapse в одиночку; шумит в высокой размерности |
| **FID (Fréchet Inception Distance)** | Расстояние между распределениями реальных и сгенерированных изображений в пространстве признаков Inception | Стандарт для оценки качества генерации **изображений**; сравнение моделей | Только изображения; нужна предобученная Inception; чувствителен к размеру выборки |
| **Inception Score (IS)** | Качество + разнообразие изображений по уверенности классификатора | Быстрая грубая оценка изображений | Только изображения; не сравнивает с реальным распределением; устарел, легко обмануть |
| **Precision / Recall для генеративных моделей** | Раздельно: качество (precision) и покрытие мод (recall) | Когда важно явно отделить «реалистичность» от «разнообразия»; диагностика mode collapse | Сложнее в реализации; зависит от пространства признаков |
| **t-SNE / UMAP** | Качественная визуализация совпадения распределений | Отчёты, диагностика mode collapse, разведка | Не численная метрика; искажает глобальную геометрию |

**Правило большого пальца:** для изображений основная численная метрика — **FID**, дополнять её **precision-recall** (качество + разнообразие отдельно) и **t-SNE** (визуальная диагностика). Для табличных данных и эмбеддингов, где Inception-сеть неприменима, — **leave-one-out 1-NN accuracy**. Inception Score сегодня считается устаревшим и как единственная метрика не используется.

## 8. Чек-лист практика

| Задача | Рекомендуемый подход |
|---|---|
| Синтетический датасет картинок для обучения / публикации | GAN (DCGAN baseline → diffusion при нехватке качества); приёмка через FID + 1-NN test |
| Аугментация редких классов в компьютерном зрении | Conditional GAN под нужный класс; проверять разнообразие (recall, t-SNE) |
| Синтетические табличные данные (транзакции, записи клиентов) | Табличный GAN (CTGAN-семейство); метрика — leave-one-out 1-NN accuracy |
| Anonymization персональных данных | GAN + обязательная проверка на утечку (1-NN accuracy НЕ должна быть сильно ниже 0.5) |
| Быстрый прототип генерации изображений 64-128 px | DCGAN — простой baseline на одной GPU |
| Фотореалистичные изображения высокого разрешения | StyleGAN2/3 или diffusion (DCGAN не тянет 512+) |
| Super-resolution, апскейл снимков/видео | GAN с adversarial loss (SRGAN-семейство) — резче, чем пиксельный loss |
| Image-to-image translation (раскраска, скетч→фото, перенос стиля) | pix2pix (парные данные) / CycleGAN (без пар) на DCGAN-скелете |
| Латентная арифметика, плавная интерполяция, редактирование атрибутов | VAE (структурированный латент) или GAN + GAN inversion |
| Кодирование объекта в латент / поиск похожих | Автоэнкодер / VAE (энкодер встроен), не чистый GAN |
| Детекция дипфейков и сгенерированного контента | Свёрточный классификатор «реальное/фейк»; регулярное дообучение под новые генераторы |
| Anomaly detection на изображениях (брак, патология) | AnoGAN-подход: GAN на нормальных данных, anomaly score = дискриминатор + ошибка реконструкции |
| Оценка качества генерации изображений | FID (основная) + precision-recall + t-SNE (визуальная диагностика) |
| Оценка генератора табличных данных / эмбеддингов | Leave-one-out 1-NN accuracy (целиться в 0.5 с обеих сторон) |
| Визуальная проверка «совпали ли распределения» | t-SNE одним `fit_transform` на объединённой выборке, фиксировать `random_state` |
| Диагностика mode collapse | t-SNE (кластеры реальных vs сжавшиеся фейки) + precision-recall |
| «Несходящийся» лосс GAN при обучении | Это норма для GAN — не лечить; проверять равновесие сетей, label smoothing, баланс learning rate |
