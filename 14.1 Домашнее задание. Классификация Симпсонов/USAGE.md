# Прикладное применение: классификация изображений, transfer learning и fine-tuning CNN

> Этот документ дополняет теоретический конспект (SUMMARY.md) практической перспективой: где и как методы из урока 14.1 (CNN, transfer learning, аугментации, WeightedRandomSampler, AdamW, LR schedulers, TTA, F1-score) применяются в реальных задачах IT/DevOps, аналитики, финансов, медицины, промышленности, ритейла и поиска аномалий.

---

## 1. Свёрточные нейросети (CNN) для image classification

### Где применяется

**Модерация контента (IT, соцсети, UGC-платформы).** Любая соцсеть (Instagram, TikTok, Reddit, VK) гоняет каждую загруженную картинку через CNN-классификатор, который решает: допустима ли картинка, содержит ли обнажёнку, насилие, оружие, спам-логотип. Маленькие специализированные CNN (MobileNetV3, EfficientNet-B0) дают inference за 10-30 мс на GPU и позволяют обрабатывать миллионы загрузок в сутки. Задача принципиально та же, что у нас с Симпсонами: мультиклассовая классификация картинок по контентным категориям.

**Контроль качества на производстве (промышленность).** Камера над конвейером снимает каждую деталь, CNN определяет класс дефекта: царапина, скол, неправильная сборка, допустимый брак. В автопроме (BMW, Tesla) CNN-модели заменили ручной визуальный контроль кузовов — скорость контроля выросла в 10 раз, пропуск дефектов снизился на 40-60%. Базовая архитектура — та же пятислойная Conv + FC или ResNet, что в задании.

**Медицинский скрининг (медицина).** Классификация снимков: рентген лёгких (норма / пневмония / туберкулёз / COVID-19), фундус-снимки глаз (стадия диабетической ретинопатии), дерматоскопия родинок (меланома / доброкачественное). Google Health, IBM Watson Health, российский Celsus — все используют CNN. Качество лучших моделей на дерматологии сравнимо с врачом-дерматологом (F1 ≈ 0.87-0.91).

**Документный OCR и классификация форм (финтех, аналитика).** Банк получает скан паспорта, водительского, договора — CNN первым делом классифицирует *тип документа*, чтобы направить его в правильный пайплайн OCR. Тот же подход — в страховых компаниях (паспорт vs СТС vs полис) и логистике (классификация накладных).

### Где НЕ стоит применять

- **Табличные данные.** CNN на таблицах из pandas работает хуже, чем gradient boosting (XGBoost, CatBoost, LightGBM). CNN не понимает семантики столбцов и тратит ёмкость на изучение ложных пространственных корреляций между колонками.
- **Маленькие датасеты (< 500 картинок на класс) без предобучения.** Обучение CNN с нуля на 100-200 примерах гарантированно даст переобучение. Выход — transfer learning, но тогда работает не «CNN как таковая», а предобученные веса.
- **Задачи, где нужна интерпретируемость по закону.** В регулируемых доменах (кредитный скоринг, медицинская диагностика для выписки диагноза) «чёрная коробка» CNN требует дополнительного объяснения (Grad-CAM, SHAP для изображений) — что удорожает проект.

### Типичные ошибки

1. **Забыть про нормализацию входа.** Картинки с пикселями 0-255 без нормализации дают градиенты огромной величины — модель расходится. Обязательно `ToDtype(torch.float32, scale=True)` + `Normalize(mean, std)`. Для pretrained-сетей — строго ImageNet mean/std.
2. **Несовпадение разрешения обучения и продакшена.** Модель обучена на 224×224, а продакшен отдаёт ей 300×400 без resize — качество резко падает. Resize/Crop должен быть частью inference-пайплайна, а не только обучающего preprocessor.
3. **Утечка тестовых картинок в train.** В датасетах с веб-скрейпинга часто одна и та же картинка лежит в обеих выборках (дубликаты по хешу или с небольшими изменениями). Модель запоминает её — валидационный F1 завышен, в продакшене — обвал. Обязательно дедупликация по perceptual hash.

## 2. Transfer learning и fine-tuning на ImageNet-моделях (ResNet и др.)

### Где применяется

**Запуск любого computer vision продукта с нуля (стартапы, R&D).** Правило почти универсальное: если есть меньше 100 000 размеченных картинок, бери предобученный на ImageNet backbone (ResNet50, EfficientNet-B3, ConvNeXt-Tiny) и дообучай. Это экономит недели обучения и даёт +10-30% к метрике по сравнению с обучением с нуля. В задании 14.1 ResNet50 с fine-tune за 6 эпох превзошёл SimpleCnn — это ровно тот сценарий, который вы увидите в любом стартапе-классификаторе.

**Медицинская визуализация (медицина).** Медицинских размеченных снимков всегда мало (разметка стоит дорого, требует врача). Transfer learning с ImageNet-ResNet на рентгене лёгких поднимает AUC с 0.72 (with nuля) до 0.88+ (fine-tune) при тех же 5000 снимках. Альтернатива — предобучение на RadImageNet (медицинский «ImageNet»), который даёт ещё лучшие начальные веса.

**Ритейл-аналитика и визуальный поиск (e-commerce).** Яндекс.Маркет, Wildberries, Lamoda используют fine-tuned ResNet/EfficientNet для: распознавания товара по фото («похожие товары»), категоризации фоток от продавцов (автоматическая проверка «это действительно платье»), поиска дублей в каталоге. Обычный подход — fine-tune на собственном каталоге + последний слой как эмбеддинг для similarity search.

**Предиктивное обслуживание по визуальной инспекции (промышленность/IoT).** Дроны облетают ветрогенераторы и ЛЭП, CNN классифицирует дефекты лопастей и изоляторов. Размеченных данных по редким дефектам — единицы сотен. Без transfer learning задача не решается, с ImageNet-fine-tune — получают production-ready модель за месяц.

### Где НЕ стоит применять

- **Домены, сильно отличающиеся от ImageNet.** Медицинские МРТ-объёмы (3D), гиперспектральные спутниковые снимки, ультразвук, рентген в grayscale — распределение пикселей сильно отличается от натуральных фотографий. Transfer с ImageNet помогает слабо. Лучше: предобучение SimCLR/MoCo на собственных нераз­меченных данных или специализированные модели (SAM-Med, RadImageNet).
- **Очень большие собственные датасеты (миллионы картинок + бюджет на GPU).** Facebook/Meta, Google обучают свои foundation-модели с нуля на собственных данных — им ImageNet не нужен. Для них transfer learning — не преимущество, а потеря контроля.
- **Строго real-time с жёсткими ограничениями по памяти (embedded, IoT-устройства).** ResNet50 (≈100 МБ) не поместится на микроконтроллер. Нужны MobileNet, SqueezeNet, quantized-модели — часто обученные с нуля или fine-tuned из предобученных мобильных чекпоинтов.

### Типичные ошибки

1. **Забыть заменить последний слой.** Импортируют ResNet50 с 1000 выходов и начинают учить на своих 42 классах — модель «учится» писать единицы в неправильные индексы. Обязательно `model.fc = nn.Linear(in_features, n_classes)`.
2. **Использовать LR от «с нуля» обучения (1e-2, 1e-3) для fine-tune.** Большой LR разрушает выученные ImageNet-веса за несколько шагов — качество падает до baseline. Правило: для fine-tune LR в 3-10 раз меньше, чем для with-scratch (`3e-4`, `1e-4`).
3. **Нормализация НЕ по ImageNet mean/std.** Pretrained-модель ожидает на входе картинки, нормализованные именно с этими константами. Любая другая нормализация (например, `[0.5, 0.5, 0.5]`) — и первые слои видят «сдвинутое» распределение, качество проседает на 5-15%.
4. **Не разморозить BatchNorm-статистику при маленьких батчах.** При fine-tune с batch=8-16 running mean/var в BN плохо оцениваются. Решение: заморозить BN (`eval_mode` на BN-слоях, `track_running_stats=False`) или увеличить batch.

## 3. WeightedRandomSampler и работа с дисбалансом классов

### Где применяется

**Fraud detection (финансы).** Мошеннические транзакции — 0.01-0.1% от всего трафика. Если батч формировать случайно, в 99% батчей не будет ни одного фрода — модель учит «всегда legit». WeightedRandomSampler (или его аналоги для PyTorch/Keras) принудительно делает в каждом батче 10-50% фрод-примеров. Это критичное улучшение качества, которое превращает недообученную модель в работающую.

**Медицинский скрининг редких патологий (медицина).** В датасете рентгенов 2% — пневмония, 98% — норма. Без балансировки модель пропускает пневмонию (recall класса 1 падает до 0.2). Балансированный сэмплер поднимает recall до 0.8-0.9 при небольшой потере precision, что клинически правильно: пропустить больного хуже, чем вызвать ложную тревогу.

**Классификация персонажей/объектов с «длинным хвостом» (модерация, ритейл).** Как в задании с Симпсонами: 2246 картинок Гомера vs 3 картинки Lionel Hutz. Аналогично в каталоге товаров Wildberries: тысячи iPhone-кейсов vs десятки редких брендов. WeightedRandomSampler даёт редким классам шанс быть «увиденными» моделью столько же раз, сколько доминирующим.

**Аномалии в логах и security-событиях (IT/DevOps).** Настоящих security-инцидентов (SQL injection, brute force, data exfiltration) — единицы на миллион запросов. Для обучения классификатора атак используют упрощённую балансировку через WeightedRandomSampler либо oversampling редких атак + undersampling нормального трафика.

### Где НЕ стоит применять

- **Когда метрика уже учитывает дисбаланс (F1-macro, AUC-PR, weighted loss).** Если loss function уже взвешенная (`CrossEntropyLoss(weight=...)`), дополнительный WeightedRandomSampler может привести к over-correction — редкие классы начнут доминировать в предсказаниях.
- **Когда дисбаланс «нативный» и должен сохраниться.** Если вы моделируете именно априорную вероятность класса (например, вероятность дефолта ~3% — и модель должна это отражать), балансировка исказит калибровку вероятностей. В production на реальных данных предсказания будут завышены в 10+ раз.
- **Очень маленькие редкие классы (< 5-10 примеров).** Oversampling 3-х картинок в миллион батчей — это не обучение, а запоминание. Модель переобучится на эти 3 конкретные картинки. Лучше: собрать больше данных, использовать few-shot learning, или объединить редкие классы в «other».

### Типичные ошибки

1. **Перепутать веса сэмплов и веса классов.** `WeightedRandomSampler` принимает вектор длины `n_samples` (вес для каждого объекта), а не длины `n_classes`. Очень частая ошибка — передать массив из 42 весов на 16746 объектов.
2. **Оставить `shuffle=True` вместе с sampler.** DataLoader ругается: `sampler option is mutually exclusive with shuffle`. Нужно либо `shuffle`, либо `sampler`, но не одновременно.
3. **Применить балансировку к валидации/тесту.** Валидационная выборка должна отражать реальное распределение — иначе метрика не соответствует тому, что увидит production. Балансировка только в train-loader.
4. **Забыть пересчитать веса при изменении train-выборки.** После cross-validation split веса надо пересчитывать на каждом фолде — иначе веса от одного фолда применяются к другому с искажённой статистикой.

## 4. Аугментации изображений

### Где применяется

**Распознавание дефектов на производстве (промышленность).** Разметка дефектов стоит дорого (инженер-контролёр смотрит каждую картинку), поэтому на класс «трещина корпуса» удаётся собрать 200-500 снимков. Аугментации (Rotation, HorizontalFlip, ColorJitter, RandomCrop, GaussianBlur) виртуально увеличивают датасет в 10-50 раз и повышают F1 с 0.75 до 0.90+. Особенно важны для дефектов, которые могут появиться в любой ориентации (царапина под любым углом).

**Медицинская визуализация (медицина).** Рентген и гистология — плоские картинки с вариабельностью освещения (разные аппараты, разная экспозиция). Аугментации `RandomBrightnessContrast`, `RandomGamma`, elastic deformation имитируют эту вариабельность и делают модель устойчивой к замене аппарата. Без аугментаций модель часто «запоминает» аппарат вместо патологии.

**Self-driving и CV для роботов (автопилот, робототехника).** Камеры на машинах/роботах работают в любых условиях: солнце, дождь, ночь, тени. Tesla Autopilot, Яндекс.Авто тренируют сети с агрессивными аугментациями освещения, погоды, шума. RandAugment и AutoAugment — автоматизированные пайплайны аугментаций, которые подбирают оптимальный набор.

**Модерация контента и распознавание лиц (IT, security).** Пользователи загружают селфи в любых условиях — с фильтрами, в низком разрешении, с артефактами JPEG. HorizontalFlip, ColorJitter, JPEG-сжатие, ResizedCrop готовят модель к этому разнообразию. В face-recognition без аугментаций точность на «диких» фото падает на 10-20%.

### Где НЕ стоит применять

- **Аугментации, меняющие семантику класса.** HorizontalFlip для распознавания букв/цифр — катастрофа (`b` → `d`, `6` → что-то бессмысленное). Для дорожных знаков-стрелок («поворот налево») — тоже. Аугментации должны сохранять метку.
- **Медицинские снимки с направленной семантикой.** ECG-сигнал, МРТ с метками «левая/правая сторона тела» — отражение может перевернуть клинический смысл. Аугментации ограничиваются тем, что физически допустимо (небольшие сдвиги, шум).
- **Когда данных уже очень много (миллионы примеров).** На 10М картинок аугментации дают маржинальный прирост (+0.5%), но замедляют обучение в 2-3 раза. В Google-scale-проектах часто обходятся минимальными (flip + crop) или вовсе без.
- **Сильные аугментации на тестовом распределении.** Если продакшен показывает модели «чистые» студийные фото (каталог одежды), обучение с `ColorJitter(hue=0.5)` добавит шум, которого в продакшене не будет — качество упадёт.

### Типичные ошибки

1. **Аугментации на валидации/тесте (в обычном пайплайне).** Валидация должна быть детерминированной: только Resize + Normalize. Случайные аугментации на val делают метрику нестабильной от запуска к запуску. Исключение — Test-Time Augmentation (отдельная техника, см. раздел 7).
2. **Слишком агрессивные параметры.** `ColorJitter(brightness=1.0, hue=0.5)` превращает картинку в неузнаваемый цветной шум — модель учится на артефактах, а не на семантике. Правило: аугментированная картинка должна быть легко узнаваема человеком как тот же класс.
3. **Порядок операций в Compose.** `Normalize` должен идти ПОСЛЕ всех пиксельных трансформаций (Resize, Crop, Flip, ColorJitter) и ДО тензорных фильтров (Cutout на уже нормализованном). Неправильный порядок — либо ошибка, либо «молчаливо» ломает обучение.
4. **Использовать PIL-аугментации на тензорах (или наоборот).** `torchvision.transforms` старого API работал только с PIL, новый `v2` — и с PIL, и с тензорами. Смешение API в одном Compose даёт невнятные ошибки типа `'Tensor' object has no attribute 'resize'`.

## 5. AdamW и weight decay

### Где применяется

**Обучение любой современной нейросети (общая практика).** AdamW — де-факто стандарт в 2022-2026 годах для обучения CNN, трансформеров, LLM, диффузионных моделей. Hugging Face, PyTorch Lightning, Keras, fast.ai — все показывают AdamW в туториалах по умолчанию. В задании 14.1 AdamW применён именно как стандарт для fine-tune ResNet50.

**Fine-tuning больших моделей (LLM, CV foundation models).** При fine-tune GPT-3.5, Llama, CLIP, ViT используется AdamW с маленьким LR (1e-5 - 3e-4) и weight_decay=0.01-0.1. Weight decay здесь критичен: без него большие модели быстро переобучаются на маленьких fine-tune-датасетах.

**Рекомендательные системы с deep learning (ритейл, IT).** Two-tower модели (user-tower + item-tower), DLRM, DeepFM — все обучаются AdamW. Weight decay регуляризует эмбеддинги пользователей/товаров, предотвращая их «распухание» до нефизичных норм.

**Обучение моделей для финансовых временных рядов (финансы).** CNN/Transformer-модели для прогноза волатильности, сигналов в HFT — там склонность к переобучению максимальная (короткие датасеты, много шума). AdamW + weight_decay 1e-3 - 1e-2 — ключевой регуляризатор.

### Где НЕ стоит применять

- **Очень большие батчи и крупномасштабное обучение.** На batch ≥ 8K AdamW хуже, чем LAMB или LARS (модификации Adam с адаптивной нормой слоя). Google использует LARS для ResNet на ImageNet с batch=32K.
- **Когда нужна воспроизводимость до бита и стабильность сходимости (некоторые RL-сценарии).** SGD с momentum более предсказуем, его траектория обучения легче анализировать. В DQN, PPO часто используют SGD или Adam (не AdamW).
- **Сильно разреженные градиенты (embedding tables с миллиардами параметров).** Для sparse-эмбеддингов лучше подходят SparseAdam, Adagrad — они эффективнее по памяти и вычислениям.

### Типичные ошибки

1. **Путать Adam и AdamW.** В Adam weight_decay реализован как L2-penalty в loss, что в сочетании с адаптивным LR работает некорректно. AdamW применяет decay напрямую к весам. Различие: AdamW всегда лучше на практике, но в старых туториалах часто фигурирует Adam.
2. **Применять weight_decay к BatchNorm и bias.** Decay должен действовать только на матрицы весов Linear/Conv. Для BN-параметров и bias decay приводит к нестабильности. Решение: в PyTorch разделять параметры на две группы (`param_groups`) с разным weight_decay.
3. **Слишком большой weight_decay (> 0.1) для fine-tune.** Агрессивный decay «стирает» предобученные веса — fine-tune перестаёт быть fine-tune. Для fine-tune: 1e-4 - 1e-3. Для with-scratch: 1e-4 - 1e-2. Для LLM: 0.01-0.1.
4. **Слишком маленький LR + большой weight_decay.** Если LR = 1e-6, а decay = 0.1, то веса «тают» быстрее, чем оптимизируются — loss не убывает. Weight decay нужно масштабировать с LR.

## 6. LR schedulers (Cosine, ReduceLROnPlateau, другие)

### Где применяется

**Обучение CNN/Transformer с нуля или fine-tune (общая практика).** Почти все SOTA-результаты на ImageNet/COCO/GLUE используют какой-то scheduler. Cosine — самый популярный для fixed-length обучения (знаем, сколько эпох), ReduceLROnPlateau — для обучения «до сходимости». В задании 14.1 CosineAnnealingLR на 6 эпохах плавно снижает LR с 3e-4 до ~1e-5, что позволяет модели сначала «широко искать», а потом «точно настроиться».

**Большие LLM: warmup + cosine decay (IT, NLP).** Стандарт обучения GPT, LLaMA, T5: первые 1-10% шагов — линейный warmup от 0 до пикового LR, потом — cosine decay до ~10% пикового. Без warmup большие модели часто расходятся на первых шагах.

**Production-модели с непрерывным переобучением (IT, ритейл).** Модели, которые переобучаются еженедельно на новых данных, обычно используют ReduceLROnPlateau: если валидационная метрика перестала улучшаться 2-3 эпохи — LR делится на 10. Это автоматическая «подстройка» без ручного тюнинга графика.

**Поисковые и ранжирующие модели (IT, e-commerce).** Deep Ranking, RankNet, трансформеры для поиска — часто обучаются на ступенчатом StepLR (reduce LR каждые N эпох в 10 раз) или на OneCycleLR (быстрый рост + медленный спад). Выбор зависит от того, насколько датасет стабилен во времени.

### Где НЕ стоит применять

- **Для классических ML-моделей с закрытыми оптимизаторами (LogReg, XGBoost).** sklearn сам управляет шагом в L-BFGS/SAG — scheduler там не применим.
- **При очень коротком обучении (1-2 эпохи).** Scheduler не успеет «отработать» — эффект минимальный. Либо увеличить epochs, либо убрать scheduler.
- **Когда валидационная метрика сильно шумная (маленький val-set).** ReduceLROnPlateau будет срабатывать на случайных флуктуациях — LR упадёт слишком рано. Решение: увеличить `patience`, использовать Cosine вместо Plateau.

### Типичные ошибки

1. **Забыть вызвать `scheduler.step()`.** Классическая ошибка: scheduler создан, но метод не вызывается в train-loop — LR не меняется. Нужно `scheduler.step()` после `optimizer.step()` (или после каждой эпохи для CosineAnnealingLR).
2. **Вызывать `.step()` в неправильном месте.** Для `CosineAnnealingLR`/`StepLR` — после эпохи. Для `OneCycleLR`/`CosineAnnealingWarmRestarts` с batch-level — после каждого батча. Для `ReduceLROnPlateau` — с аргументом (val_loss или val_f1).
3. **T_max ≠ числу эпох в CosineAnnealingLR.** Если `T_max=10`, а обучение идёт 6 эпох — LR не успеет спуститься до минимума, обучение закончится на LR ~50% пикового. Либо `T_max=n_epochs`, либо смириться.
4. **Сочетание scheduler + warmup без library-helper'а.** Ручная склейка warmup и cosine — источник багов. Лучше использовать `transformers.get_cosine_schedule_with_warmup` или `torch.optim.lr_scheduler.SequentialLR`.

## 7. Test-Time Augmentation (TTA)

### Где применяется

**Kaggle-соревнования по image classification (ML research, общая практика).** TTA — стандартный трюк для выжимания последних долей процента метрики. На Kaggle-конкурсах CV-тематик (Cassava Leaf, Plant Pathology, SIIM-ISIC Melanoma) топовые решения почти всегда включают TTA из 5-10 трансформаций. В задании 14.1 применён минимальный TTA (original + horizontal flip), типичный прирост — +0.5-1.5% F1.

**Медицинская диагностика в продакшене (медицина).** Когда от предсказания зависит жизнь пациента (скрининг онкологии), допустимо заплатить в 5-10 раз больше вычислений, чтобы получить +1-2% к AUC. Радиологические AI-системы (Aidoc, Zebra Medical) применяют TTA с 4-8 вариантами поворотов и отражений на каждый снимок.

**Production с высокой ценой ошибки (промышленность, security).** Автоматическое обнаружение дефектов в критичных отраслях (авиастроение, атомная энергетика) — тот случай, когда каждая пропущенная трещина может стоить миллиарды. TTA + ансамбль моделей — обязательный паттерн.

**Видео-классификация с multi-crop (модерация контента).** Для классификации видеороликов часто берут 3-5 кропов разной временной позиции, усредняют предсказания — это форма TTA на временной оси. Применяется в YouTube Content ID, TikTok moderation.

### Где НЕ стоит применять

- **Real-time-системы с жёстким SLA.** TTA увеличивает inference-время в N раз (N — число аугментаций). Для fraud detection с SLA < 50 мс TTA обычно невозможна.
- **Мобильные/edge-устройства.** Батарея и тепловыделение — TTA удваивает/утраивает потребление. На iPhone или умной камере TTA редко оправдана.
- **Когда base-модель уже насыщена (метрика на plateau).** Если F1 = 0.99, TTA даст +0.001 — не стоит усложнения пайплайна.
- **Неподходящие аугментации.** HorizontalFlip для OCR, вертикальный поворот для медицинских снимков с латеральностью — TTA с такими трансформациями УХУДШИТ предсказания.

### Типичные ошибки

1. **Усреднять логиты вместо softmax-вероятностей.** Логиты — в несоизмеримых шкалах до softmax. Корректно: `mean(softmax(logits))`, а не `softmax(mean(logits))`. Вторая формула тоже работает, но она арифметически другая и иногда даёт заметно другие результаты.
2. **Использовать разную нормализацию на train/TTA.** Если на train была ImageNet mean/std, а в TTA забыли её применить — модель получит сдвинутое распределение и предсказания будут мусор.
3. **TTA только на одну трансформацию.** +0.3% маловато — если уж пошли на TTA, брать 4-8 трансформаций (h-flip, multi-crop, лёгкие повороты). Стоимость та же, выигрыш больше.
4. **Забыть отключить Dropout/BatchNorm-обучение.** TTA делается в режиме без обучения. Если модель случайно осталась в train-режиме — dropout будет давать разные предсказания на одинаковом входе, усреднение сведёт их в шум.

## 8. F1-score micro vs macro — выбор метрики под задачу

### Где применяется

**Модерация контента с сильным дисбалансом классов (IT, соцсети).** Если 90% контента — «нормально», а 10% — несколько категорий нарушений, micro-F1 покажет в основном качество на «нормально» (доминирующий класс), а macro-F1 — среднее качество по всем категориям нарушений. Бизнесу важнее именно macro-F1: все категории должны ловиться.

**Многоклассовая классификация с примерно равными классами (задание 14.1 — Симпсоны).** Здесь дисбаланс есть (2246 vs 3), но бизнес-задача Kaggle — «сколько процентов тестовых картинок угадано». Это в точности micro-F1. Выбор метрики диктуется не дисбалансом, а постановкой: один класс = одна картинка, цена ошибки одинакова.

**Медицинская классификация с редкими патологиями (медицина).** Диагностика редких заболеваний: если 99% снимков — норма, 1% — редкая патология, macro-F1 — единственная честная метрика. Micro-F1 = 0.99 даже у модели «всегда говорим норма», macro-F1 = 0.50, что сразу показывает бесполезность.

**Классификация документов по типам (финтех, enterprise).** У банка 50 типов документов, от паспорта (миллионы) до редкого доверительного акта (тысячи). Macro-F1 показывает, хорошо ли работает модель на всех типах, micro-F1 — на «массовом» потоке.

### Где НЕ стоит применять

- **Бинарная классификация.** В двухклассовом случае «classical F1» (без micro/macro) обычно считается для положительного класса — это стандарт. Micro-F1 в бинарной задаче равен accuracy.
- **Регрессия.** F1 — метрика только для классификации. Для регрессии используйте MAE, RMSE, MAPE, R².
- **Ранжирование и retrieval.** F1 игнорирует порядок предсказаний. Для ранжирования — NDCG, MAP, Precision@k.
- **Multi-label classification без понимания разницы.** В multi-label задачах micro/macro F1 считаются по-другому (per-class binary). Применять наивно — получить неверные цифры.

### Типичные ошибки

1. **Использовать micro-F1 на сильно несбалансированных данных как финальную метрику.** Micro-F1 ≈ accuracy в мультиклассе, плохо отражает качество на редких классах. Для дисбаланса нужен macro-F1 или weighted-F1.
2. **Сравнивать macro-F1 моделей на разных наборах классов.** Если добавили новый класс — macro-F1 изменится не из-за качества, а из-за изменения знаменателя. Нужно либо переобучить обе модели на одинаковом наборе, либо сравнивать per-class.
3. **Путать weighted-F1 и micro-F1.** weighted-F1 — среднее F1 по классам, взвешенное по размеру класса. Micro-F1 — глобальный F1 по всем предсказаниям. На несбалансированных данных они различаются на проценты и могут вести к разным выводам.
4. **Считать F1 без учёта zero-division.** Для пустого класса (0 предсказаний данного класса) F1 не определён. `sklearn` по умолчанию выдаёт warning, но если просто усреднять — легко получить NaN или ошибку. Использовать `zero_division=0`.

## 9. Сравнение с альтернативами: decision framework

### Когда выбирать Transfer Learning с ResNet

| Ситуация | Почему ResNet50 fine-tune |
|---------|--------------------------|
| 1K-100K размеченных картинок | Sweet spot transfer learning: ImageNet даёт sweet spot инициализацию, ваших данных достаточно для fine-tune |
| Картинки похожи на ImageNet (естественные фото) | Pretrained-веса переносятся отлично — получаем +10-20% к качеству |
| Есть GPU, но нет Google-scale кластера | ResNet50 обучается за часы на одной GPU, помещается в память 8-16 ГБ |
| Задача: classification, detection backbone, feature extraction | Архитектура ResNet универсальна под все эти задачи |
| Нужна «скучная надёжная» модель для production | ResNet — проверенная 10 лет, стабильная, предсказуемая |

### Когда обучать CNN с нуля

| Ситуация | Почему with-scratch |
|---------|---------------------|
| Очень большой собственный датасет (10M+ картинок) | ImageNet (1.2M) меньше вашего — fine-tune теряет смысл |
| Домен далёк от ImageNet (медицинские, спутниковые, радарные) | ImageNet-фильтры неэффективны, лучше выучить свои |
| Нужна нестандартная архитектура под специфику задачи | Например, 3D-CNN для МРТ, temporal CNN для ECG |
| Цель — research, публикация | Новые архитектуры обычно обучают с нуля для «чистого» сравнения |
| Ограничения по памяти/latency диктуют кастомный дизайн | MobileNet-клон под конкретное железо |

### Когда EfficientNet vs ResNet vs ViT vs MobileNet

| Модель | Размер | Когда выбрать | Ограничения |
|--------|--------|---------------|-------------|
| **ResNet50** | ~100 МБ, 25M params | Baseline, production, проверенная надёжность | Есть более точные альтернативы при том же компьюте |
| **EfficientNet-B3/B4** | ~48/75 МБ, 12/19M params | Лучшее качество/компьют, mobile-friendly | Сложнее обучать, чувствительнее к гиперпараметрам |
| **ViT (Vision Transformer)** | ~330 МБ, 86M params | Максимум качества на больших датасетах, multimodal pipelines | Требует много данных или сильной предобученности, медленнее на маленьких картинках |
| **ConvNeXt** | ~200 МБ, 50M params | Современный CNN с transformer-like трюками, SOTA на ImageNet | Сравнительно новый, меньше экосистема |
| **MobileNetV3** | ~20 МБ, 5M params | Edge-устройства, мобильные приложения, real-time | Качество ниже, чем у «большой» модели |
| **Swin Transformer** | ~100 МБ, 28M params | Detection/segmentation, иерархические фичи | Сложнее интеграция, чем у ResNet |

**Правило большого пальца:** начни с ResNet50 fine-tune. Если качество устраивает — не меняй. Если нужно больше качества и есть компьют — EfficientNet-B3 или ViT-B/16. Если нужно меньше размер — MobileNetV3 или дистилляция ResNet-а.

### Transfer learning vs обучение с нуля: решение

```
1. Сколько размеченных картинок у тебя?
   < 1K  -> Transfer learning обязателен. Возможно feature extractor (заморозить backbone).
   1K-100K -> Transfer learning (fine-tune) почти всегда выигрывает.
   100K-10M -> Transfer learning ещё помогает, но with-scratch возможно.
   > 10M -> With-scratch конкурентен с transfer learning.

2. Насколько твой домен похож на ImageNet?
   Естественные фото -> Transfer learning идеален.
   Медицина/спутники/радары -> Transfer с ImageNet менее эффективен, рассмотри self-supervised pretraining.

3. Какой у тебя бюджет (GPU + время)?
   Минимальный (1 GPU, 1 день) -> Transfer learning, fine-tune маленькой модели.
   Средний (1 GPU, недели) -> Transfer с большой моделью или with-scratch с маленькой.
   Большой -> With-scratch возможен, но обычно transfer всё равно выигрывает по качеству/время.
```

## 10. Чек-лист практика

| Задача | Рекомендуемый подход |
|--------|---------------------|
| Классификация изображений, датасет 1K-100K | ResNet50 fine-tune + аугментации + AdamW + CosineLR |
| Image classification на edge/mobile | MobileNetV3 или дистилляция из ResNet, quantization |
| Медицинская диагностика по снимкам (рентген, дерматоскопия) | RadImageNet/SimCLR pretrain -> fine-tune ResNet50 + TTA + macro-F1 |
| Контроль качества на производстве (дефектоскопия) | ResNet50/EfficientNet-B3 fine-tune + сильные аугментации (Rotation, Flip, ColorJitter) + WeightedRandomSampler для редких дефектов |
| Модерация UGC-контента (соцсети) | EfficientNet-B0/MobileNetV3 fine-tune, macro-F1 как метрика, TTA только для граничных случаев |
| Документный OCR / классификация форм | ResNet18/50 fine-tune на аугментированных сканах, class-balanced sampling |
| Сильный дисбаланс классов (1:100 и больше) | WeightedRandomSampler + weighted CrossEntropyLoss + macro-F1 / AUC-PR |
| Мало данных на некоторые классы (редкие) | Transfer learning + агрессивные аугментации + WeightedRandomSampler + label smoothing |
| Максимум качества на Kaggle-соревновании | Ансамбль ResNet + EfficientNet + ViT, TTA с 5-10 трансформациями, Pseudo-labeling |
| Real-time inference (< 50 мс, без GPU) | MobileNetV3 / EfficientNet-Lite, без TTA, quantization INT8 |
| Visual search в ритейле (каталог товаров) | ResNet50 fine-tune, последний слой как эмбеддинг, FAISS/HNSW для поиска похожих |
| Видео-классификация | 3D-CNN или 2D-CNN + temporal pooling, TTA по временным кропам |
| Fine-tuning LLM/Foundation model | AdamW с lr 1e-5 - 3e-4, weight_decay 0.01-0.1, warmup + cosine decay |
| Оценка модели при несбалансированных классах | macro-F1 или weighted-F1 + classification_report для per-class анализа |
| Оценка модели при равных классах | accuracy или micro-F1 (они равны в multiclass) |
| Production-переобучение на новых данных | ReduceLROnPlateau + early stopping по val-метрике + сохранение best checkpoint |
| Быстрый baseline для нового CV-проекта | ResNet50 + ImageNet weights + fc заменить + 5 эпох AdamW(lr=3e-4) + TTA(flip) |
