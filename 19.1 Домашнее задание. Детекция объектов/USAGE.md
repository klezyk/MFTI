# Прикладное применение: детекция объектов

> Этот документ дополняет теоретический конспект (SUMMARY.md) практической перспективой: где и как методы object detection из задания 19.1 (Backbone+FPN+Decoupled Head, TAL, DIoU, NMS, аугментации) применяются в реальных задачах IT, ритейла, промышленности, медицины и автономного транспорта.

## 1. Object Detection в продакшене: выбор архитектуры и трюков

### Anchor-based vs anchor-free детекторы — где какой выбирать

**Anchor-based (Faster R-CNN, RetinaNet, YOLOv3-v5, базовый детектор из задания).** Якоря — это «шаблоны» ббоксов разного размера и пропорций, которые модель «двигает» к ground truth. Это даёт хороший inductive bias: модели проще обучаться, потому что не надо предсказывать координаты «с нуля». В продакшене anchor-based выбирают, когда:

- **Размеры объектов известны заранее и распределены ограниченно.** Видеонаблюдение в магазине: люди примерно одного роста относительно камеры, тележки — стандартного размера. Подбор `anchor_sizes=(32, 64, 128)` под распределение GT даёт прирост mAP без архитектурных изменений.
- **Нужна стабильность на маленьких датасетах (<5K изображений).** Anchor priors — это «знание» о распределении ббоксов, которое экономит обучающие примеры. На 1K картинок anchor-free детектор скорее всего переобучится, anchor-based — нет.
- **Производственный конвейер с нормированными изображениями.** Контроль качества на заводе: камера зафиксирована, объекты приезжают по конвейеру в одном и том же ракурсе. Якоря под три размера дефекта работают идеально.

**Anchor-free (FCOS, CenterNet, YOLOX, YOLOv8/v11).** Модель предсказывает координаты ббокса напрямую от центра пикселя (или от ключевой точки). Плюсы — нет ручной настройки якорей, меньше гиперпараметров, лучше работает на объектах с экстремальными aspect ratio (длинные надписи, узкие столбы, медицинские артефакты). Anchor-free выбирают, когда:

- **Распределение размеров объектов сильно разнородное.** Аэрофотосъёмка: на одном кадре машины 20px и здания 500px — anchor priors на три размера тут не помогут.
- **Современный production без legacy-инфраструктуры.** Если стартуете проект в 2025 — берите YOLOv8/v11 или RT-DETR. Anchor-free у них лучше отлажены и обучены на огромных корпусах.
- **Объекты с непредсказуемой геометрией.** OCR (тексты), детекция трещин в инфраструктуре, поиск патологий — здесь якоря только мешают.

**Практический выбор.** Если у вас Halo-подобная задача (фиксированные классы, ограниченная съёмка) — anchor-based с правильно подобранными якорями даст хороший mAP быстрее. Если речь про общую задачу с непредсказуемыми объектами — берите anchor-free. В индустрии последние 2-3 года тренд явно сдвигается к anchor-free, потому что инженерные затраты на тюнинг якорей часто выше выгоды.

### Transfer learning + fine-tune backbone'а — типичные сценарии

В задании 19.1 backbone — `efficientnet_b0` с pretrained ImageNet-весами и опциональной разморозкой `k` последних слоёв. Это не учебный приём, а стандарт индустрии. Обучать детектор с нуля на 600 картинках Halo — заведомо проиграть.

**Сценарий 1: домен близок к ImageNet (фотографии реального мира).** Видеонаблюдение, retail-аналитика, детекция животных — backbone разморозить полностью или последние 2-3 блока, learning rate для backbone — в 10-100 раз меньше, чем для head. Типичная схема: `lr_head=1e-3`, `lr_backbone=1e-4`, AdamW с weight_decay=1e-4. За 20-50 эпох выходим на боевые метрики.

**Сценарий 2: домен далёк от ImageNet (медицина, сателлиты, рентген).** Заморозить backbone бесполезно — фичи pretrained сети не подходят. Нужно либо разморозить всё с самого начала (lr_backbone=1e-4, осторожно), либо найти pretrained веса под конкретный домен (RadImageNet для медицины, BigEarthNet для сателлитов). Иногда полезен self-supervised pre-train (DINO, MAE) на собственных неразмеченных данных перед детекцией.

**Сценарий 3: production с инкрементным дообучением.** Модель работает в проде, периодически приходят новые размеченные данные. Полное переобучение дорогое. Стратегия: заморозить backbone, дообучать только neck+head на новых данных, раз в квартал — полный warm-restart с разморозкой всего. Так делают, например, в системах модерации UGC-контента.

**Антипаттерн.** Обучать ResNet50 с нуля на 1K картинок «потому что данные специфичные». Это даст хуже метрики, чем pretrained ImageNet-веса, даже если домен сильно отличается — низкоуровневые фильтры (края, текстуры) универсальны.

### FPN и PANet — где multi-scale критичен

FPN (Feature Pyramid Network) и его расширение PANet — это про детекцию объектов разного размера. Базовый детектор из семинара использует один уровень фичей с одним reduction (например, /32). Это значит, что мелкие объекты (10-20 пикселей) на нём детектируются плохо: после 5 даунсемплингов они превращаются в 0.3-0.6 пикселя на feature map.

**Где multi-scale критичен:**

- **Видеонаблюдение и retail analytics.** Камера снимает зал магазина. Ближе к камере человек занимает 200×400 пикселей, в дальнем углу — 30×60. Без FPN модель ловит только ближних, дальних теряет. С FPN на 3-4 уровнях каждый размер обрабатывается на «своём» уровне иерархии — feature map с большим разрешением для мелких, с маленьким — для крупных.
- **Аэрофотосъёмка и дроны.** Машины 15px, дома 300px на одном кадре. Без multi-scale сеть либо учится на крупных и пропускает мелкие, либо наоборот. FPN+PANet — стандарт в DOTA-датасете и аналогичных задачах.
- **Контроль качества: мелкие дефекты на больших изделиях.** Микротрещины 5-10 пикселей на изображении детали 1024×1024. Только высокоразрешённый уровень FPN их вытащит.
- **Автономный транспорт.** Пешеход за 100 метров — 20 пикселей. Машина в 5 метрах — 800 пикселей. Без multi-scale ADAS-система слепа к одной из этих категорий.

**PANet vs FPN.** PANet добавляет bottom-up путь поверх top-down пути FPN. Это даёт лучшее smешивание low-level (текстуры, края) и high-level (семантика) фичей. На практике PANet даёт +1-2 mAP по сравнению с FPN при тех же параметрах. Если железо позволяет — берите PANet (он встроен в YOLOv5+, YOLOv8+).

**Когда можно обойтись без FPN.** Если все объекты примерно одного размера (специфичный production line, документы со стандартным расположением полей) — один уровень фичей подобранного reduction даст не хуже.

### Decoupled Head — когда задачи cls и bbox конфликтуют

Базовая голова детектора шарит свёртки между классификацией и регрессией bbox. Это эффективно по параметрам, но эмпирически (YOLOX, 2021) даёт хуже метрики: классификация хочет «инвариантные к расположению» фичи (что это за объект?), регрессия хочет «локализационно-чувствительные» (где именно его край?). Decoupled Head разделяет ветки: отдельные свёртки для cls, отдельные — для bbox.

**Где это даёт максимальный профит:**

- **Объекты с похожей внешностью, но разной локализацией.** Конвейер с почти идентичными деталями: классы определяются мелкими отличиями, bbox — точными краями. Shared head конфликтует, decoupled — нет.
- **Многоклассовая детекция с тонкими отличиями.** Halo-датасет: enemy vs friendly отличаются цветом униформы, но bbox одинаков по структуре (человек в полный рост). Decoupled head даёт +1-3 mAP.
- **OCR и детекция текста.** Регрессия требует пиксель-точных границ символов, классификация — семантики (буква/цифра/символ). Полностью разные задачи в одной голове.

**Когда не нужно.** Бинарная детекция одного класса (есть/нет дефект) — конфликт минимален, shared head экономит параметры. Также для очень маленьких моделей (real-time на edge), где каждый параметр на счету.

### TAL и динамические assigner'ы — когда полезно

Базовый IoU-assigner (как `assign_target` из семинара) присваивает якорям GT по фиксированному порогу IoU (0.5-0.7). Это статическая логика: якорь либо «положительный», либо «отрицательный», независимо от того, что предсказывает модель. Проблемы:

1. На ранних эпохах большинство якорей имеют низкий IoU с GT — мало positives, обучение медленное.
2. Якорь может быть в правильном месте, но классификатор предсказывает не тот класс — он всё равно считается positive, что путает модель.
3. Один и тот же якорь может быть «лучшим» для нескольких GT — конфликт.

TAL (Task Alignment Learning, статья TOOD) и аналоги (SimOTA в YOLOX, ATSS) решают это динамически: метрика выравнивания `t = s^α · u^β` объединяет classification score и IoU. Якорь становится positive, только если он одновременно хорошо локализован И уверенно классифицируется. Это синхронизирует обучение cls и reg голов.

**Где TAL даёт максимальный профит:**

- **Сильный дисбаланс классов.** В датасете много background и мало foreground. TAL увеличивает effective positives, не «размывая» обучение слабыми кандидатами.
- **Many-to-one конфликты.** Когда несколько GT близко друг к другу (толпа людей, скопление машин) и якоря пересекаются с несколькими целями. TAL разводит конфликты по `t`-метрике.
- **Production-системы с постоянным дообучением.** Динамический assigner адаптируется к смещению распределения, статический — нет.

**Когда не нужно.** Простые задачи с разреженными объектами (один-два per image) и хорошо подобранными якорями — TAL не даёт прироста, только усложняет код. Также на этапе быстрого прототипа: сначала запустите baseline с IoU-assigner, потом подключайте TAL.

### DIoU/CIoU loss — когда переходить с SmoothL1

SmoothL1 (он же Huber loss) учит регрессировать смещения покоординатно: tx, ty, tw, th — четыре независимых регрессии. Проблема: лосс не знает про геометрию. Два ббокса с одинаковым SmoothL1 могут иметь IoU=0.8 и IoU=0.3.

DIoU/CIoU — IoU-based losses, которые напрямую оптимизируют overlap:

- **IoU loss** = 1 − IoU. Минимум: ббоксы совпадают.
- **DIoU loss** = 1 − IoU + d²/c² (расстояние между центрами, нормированное на диагональ выпуклой оболочки). Учит модель сводить центры даже когда нет overlap.
- **CIoU loss** добавляет ещё штраф за aspect ratio. Современный стандарт в YOLOv5+/YOLOv8+.

**Когда переходить:**

- **Нужны точные ббоксы.** Медицина (хирургические маркеры), AR-фильтры, OCR — здесь критична пиксель-точная локализация. CIoU даёт +2-4 mAP@0.75 по сравнению со SmoothL1.
- **Метрика — mAP@0.5:0.95 (COCO style).** Эта метрика штрафует неточные ббоксы. SmoothL1 оптимизирует по координатам, не по IoU — несоответствие метрике обучения и метрике оценки.
- **Объекты с сильно разными размерами.** SmoothL1 одинаково штрафует ошибку 5 пикселей и для маленького, и для большого объекта. Для маленького 5px — катастрофа (IoU обвалится), для большого — мелочь. IoU-based losses учитывают это автоматически.

**Когда SmoothL1 ещё ок.** Bbox-only задачи на крупных объектах (документы, регионы интереса), где порог IoU=0.5 — главная метрика. SmoothL1 проще и стабильнее на ранних эпохах. Хорошая стратегия — стартовать со SmoothL1, переключиться на CIoU после warm-up.

### NMS / SoftNMS / Weighted Boxes Fusion — какой выбрать

Постпроцессинг: модель выдаёт сотни-тысячи candidate ббоксов, надо оставить только финальные. Алгоритмы:

**Стандартный NMS.** Сортирует по score, жадно удаляет всё, что пересекается с самым уверенным предсказанием по IoU > threshold (типично 0.5). Просто, быстро, в torchvision из коробки. Минусы: если два реальных объекта пересекаются (две стоящие машины, две головы в толпе), один из них будет «съеден». Также чувствителен к пороговому значению.

**SoftNMS.** Вместо «удалить» уменьшает score пересекающихся боксов пропорционально IoU. Боксы плотных кластеров не теряются, а получают пониженный priority. Прирост mAP на crowded scenes (толпы людей, плотные машины) — 1-2 пункта без переобучения. Реализация — несколько строк, часто берут готовую.

**Weighted Boxes Fusion (WBF).** Не выбирает «лучший» бокс, а усредняет координаты пересекающихся боксов с весами по score. Стандарт для ансамблей моделей в Kaggle. Если у вас одна модель — выгода маленькая (~0.5 mAP), если ансамбль из 3-5 — обязательно (+2-4 mAP).

**Class-agnostic NMS.** Применяет NMS не по классам, а ко всем боксам сразу. Полезно, когда модель предсказывает один и тот же объект разными классами (например, «enemy» и «enemy-head» на одной голове). Спасает от дубликатов.

**Практический выбор:**

| Задача | Рекомендация |
|---|---|
| Real-time, разреженные объекты | Стандартный NMS, threshold=0.5 |
| Crowded scenes (толпы, движение) | SoftNMS |
| Ансамбль моделей (Kaggle / production hi-quality) | Weighted Boxes Fusion |
| Multi-class конфликты (модель путает похожие классы на одном объекте) | Class-agnostic NMS |
| Точность критична, скорость не важна (медицина) | SoftNMS + post-process по доменным правилам |

### Аугментации с bbox (albumentations) — обязательные приёмы

Особенность детекции: при аугментациях надо синхронно трансформировать и изображение, и ббоксы. Поэтому стандартный `torchvision.transforms` не подходит — он не знает про bbox. Используют `albumentations` с `bbox_params=A.BboxParams(format='coco', label_fields=['labels'])`, который автоматически пересчитывает ббоксы при кропе/повороте/флипе и удаляет их, если они «вышли» за пределы кадра.

**Обязательный минимум для object detection:**

- **HorizontalFlip(p=0.5)** — почти всегда (исключение: если у объекта есть «лево/право» в семантике, например, дорожные знаки «поворот налево»).
- **RandomResizedCrop / RandomScale** — учит модель работать с разными масштабами. Особенно важно, если в трейне объекты в основном крупные, а в проде встречаются мелкие.
- **Normalize** с ImageNet mean/std — обязательно при использовании pretrained backbone.

**Сильно полезные:**

- **ColorJitter / RandomBrightnessContrast / HueSaturationValue** — устойчивость к освещению. Критично для outdoor-задач (дроны, автономный транспорт), retail (разные магазины — разное освещение).
- **GaussNoise / GaussianBlur / MotionBlur** — устойчивость к плохой камере. Обязательно для embedded/edge-устройств с дешёвой оптикой.
- **CoarseDropout (Cutout)** — случайно зануляет квадраты на изображении. Учит модель не полагаться на одну часть объекта (если закрыт лицом — определять по одежде).

**Продвинутые (домен-зависимые):**

- **Mosaic (4 изображения в одном) и MixUp** — стандарт YOLOv4+. Сильно расширяет разнообразие сцен и распределение размеров. Особенно эффективны на маленьких датасетах.
- **CutMix** — вырезает регион из одного изображения, вставляет в другое. Полезно для редких классов.
- **Copy-Paste augmentation** — копирует объекты с маской из одних кадров в другие. Лучшая аугментация для long-tail распределений (редкие классы).

**Опасные аугментации:**

- **RandomRotation** на больших углах (>15°). Если ббокс не вращается вместе с объектом (а в стандартных детекторах он остаётся axis-aligned), повёрнутый объект «вылезает» за свой ббокс. Лучше использовать осторожно или переходить на rotated-bbox задачи (DOTA-style).
- **VerticalFlip** — только если в задаче нет «верха/низа» (медицинские срезы, текстуры). Для людей/машин/животных это ломает семантику.

### mAP как KPI — как интерпретировать на практике

mAP — стандарт для отчёта в детекции, но для бизнеса нужно понимать его особенности:

**mAP@0.5 (PASCAL VOC style).** Считает predict как correct, если IoU с GT > 0.5. Метрика «модель примерно нашла объект». В бизнесе подходит для:
- Подсчёта людей/машин (не важна точная граница, важно посчитать).
- Pre-labeling датасетов (человек потом подкорректирует ббокс).
- Триггера событий (перешёл человек линию или нет — порог 0.5 ок).

**mAP@0.75 / mAP@0.5:0.95 (COCO style).** Усреднение по 10 порогам IoU от 0.5 до 0.95. Метрика «модель нашла объект И точно нарисовала границу». Подходит для:
- AR-фильтров (граница лица должна быть пиксель-точной).
- Медицинской детекции (хирургический маркер, фокус биопсии).
- OCR (точные границы текста перед распознаванием).

**Какой порог IoU выбирать для бизнеса.** Зависит от того, что делается с ббоксом дальше:
- Бокс — финальный ответ для пользователя (рамка вокруг лица в фотоприложении): IoU=0.7+ (пользователь заметит небрежность).
- Бокс — вход для следующей модели (классификатор детали, OCR): IoU=0.5 ок, последующая модель устойчива к небольшим неточностям.
- Бокс — вход для трекера или счётчика: IoU=0.5, трекер сглаживает шум.

**Подводные камни mAP:**

- **mAP агрегирует по классам.** Если у вас 4 класса (Halo: enemy/enemy-head/friendly/friendly-head) и mAP=0.5, это может быть и (0.8, 0.8, 0.2, 0.2), и (0.5, 0.5, 0.5, 0.5). Для бизнеса всегда смотрите per-class AP.
- **mAP не учитывает confidence calibration.** Модель может иметь mAP=0.7 при threshold=0.5 и mAP=0.7 при threshold=0.1 — но в первом случае пропускает половину объектов. Бизнес-метрики (precision/recall на конкретном пороге) важнее общего mAP.
- **mAP игнорирует false negatives на пустых изображениях.** Если на кадре нет объектов и модель ничего не предсказала — это +0 в mAP. Но в production это важно (вызовет ли модель false alarm в пустой комнате?).

## 2. Прикладные сценарии

### Видеонаблюдение и retail analytics

В торговых сетях (Auchan, Магнит, X5) детекция людей — фундамент аналитики посетительского трафика. Камеры на входе считают входящих/выходящих, камеры в зале строят heatmap зон с высокой плотностью покупателей, очередь на кассе детектируется и при превышении порога вызывает дополнительного кассира.

Технически: детектор класса person (часто только одного класса) запускается на 5-15 fps, дальше треккер (DeepSORT, ByteTrack) ассоциирует детекции между кадрами. Из этого получают:
- **Footfall counters.** Сколько посетителей за час. Бизнес-метрика #1 для retail.
- **Dwell time.** Сколько посетитель провёл у конкретной полки. Полезно для оптимизации выкладки.
- **Queue length.** Количество людей в очереди и time-in-queue. Триггер для оперативных решений.
- **Conversion rate.** Сколько вошедших дошли до кассы. KPI магазина.

В этом сценарии mAP@0.5 на порядок важнее mAP@0.5:0.95 — бизнесу важно «есть/нет человек», а не пиксельная точность. Аугментации обязательны: RandomBrightnessContrast (магазины с разным освещением), HorizontalFlip, мелкий MotionBlur (камеры с низкой скоростью затвора). Backbone — обычно ResNet50 или MobileNetV3 (чтобы крутиться на embedded NVR-устройствах).

В России отдельная специфика — privacy-by-design: детектор не должен сохранять лица или строить идентификацию. Поэтому архитектуры детектируют bounding box, после чего downsample/blur лица перед сохранением фрейма. Российская разработка ВТБ/Wildberries часто использует YOLOv8 + ByteTrack как production-стандарт.

### Контроль качества на производстве (дефекты на конвейере)

Промышленный QA — ключевой кейс object detection в B2B. Камера над конвейером, модель детектирует дефекты (царапины, сколы, неправильная сборка) в реальном времени. Брак отсеивается роботом-манипулятором или сигналом оператору.

Особенности:
- **Жёсткие требования к recall.** Пропустить дефект — вернуть товар клиенту, репутационный ущерб. Обычно бизнес ставит recall=0.99+ при допустимом precision 0.7-0.9 (false alarm — недорого, оператор перепроверит).
- **Сильный дисбаланс классов.** Дефекты — редкие события (0.1-1% от всей продукции). Для обучения собирают «синтетический» датасет: вручную создают дефекты или используют GAN-аугментации.
- **Контролируемые условия съёмки.** Освещение, ракурс, фон — фиксированы. Это позволяет использовать простые архитектуры (RetinaNet, YOLOv5-small) и обучать на 500-2000 примерах.
- **Real-time constraint.** Конвейер не ждёт. Инференс должен укладываться в время прохождения детали через зону камеры (50-200 мс).

Технически часто используют ResNet50+FPN или YOLOv8-nano. Decoupled Head даёт прирост, потому что классификация дефектов (царапина vs скол vs пятно) — отдельная задача от локализации. Аугментации: ColorJitter (имитация изменения освещения), небольшой RandomRotation, GaussNoise. CIoU обязателен — точные границы дефектов важны для измерения площади поражения.

Например, на сборочных линиях электроники (Foxconn, российский «Микрон») детектируют отсутствующие пайки, неправильно установленные компоненты, царапины на чипах. На пищевой промышленности (Сафмар, Ozon Food) — посторонние предметы, повреждения упаковки, отклонения в весе по визуальной оценке.

### Автономный транспорт (детекция пешеходов/машин)

ADAS (Advanced Driver Assistance Systems) и self-driving — самая сложная и регулируемая область object detection. Tesla Autopilot, Mobileye, Yandex SDG — все используют детекцию как основу восприятия.

Что детектируют:
- **Vulnerable road users.** Пешеходы, велосипедисты, мотоциклисты. Recall критичен — пропуск пешехода при переходе = летальный исход.
- **Vehicles.** Легковые, грузовики, спецтехника, эмбулансы. Классы определяют поведение: за грузовиком держим больше дистанции.
- **Traffic infrastructure.** Светофоры, знаки, разметка, шлагбаумы.
- **Construction zones.** Конусы, баррикады, рабочие в светоотражающей форме.

Особенности архитектур:
- **Multi-scale обязателен (FPN+PANet).** Пешеход за 5 метров = 400 пикселей, за 100 метров = 20 пикселей. Без multi-scale half далёких пешеходов теряется.
- **Temporal information.** Используют видеопоток, не отдельные кадры. Архитектуры типа CenterTrack, SlowFast, ВLAS-tower комбинируют детекцию с трекингом.
- **Sensor fusion.** Камера + LIDAR + radar. Детекция в каждой модальности отдельно, fusion на уровне ббоксов через WBF или learned fusion.
- **Edge inference.** GPU в машине ограничен. Используют квантование INT8, TensorRT, OpenVINO. YOLO-семейство популярно из-за оптимизации под edge.

Обучающие датасеты огромные (KITTI, nuScenes, BDD100K, Waymo Open Dataset) — миллионы аннотированных кадров. Для русских условий есть Yandex SDG и Cognitive Pilot датасеты. Аугментации специфичные: имитация дождя/снега/тумана (RainAugmenter, FogAugmenter), вариации времени суток, разные ракурсы камеры.

Ключевая метрика — не mAP, а Time-to-Collision (TTC) и Distance Estimation Error. mAP — внутренняя инженерная метрика, но регулятор (UNECE WP.29) смотрит на безопасность кейсов в реальных условиях.

### Медицина (детекция патологий на снимках)

Медицинская визуализация — рентген, КТ, МРТ, маммография, дерматоскопия — массовая область применения object detection. В России активно развивают «Цифровая медицина» (Сколково, Третье мнение), Botkin.AI для маммографии, СберМедИИ для КТ-лёгких.

Задачи:
- **Детекция узелков на КТ лёгких.** Рак лёгких на ранней стадии — узелки 5-15 мм. Радиолог пропускает 20-30% мелких узелков из-за усталости. AI находит, флагует, радиолог проверяет.
- **Маммография: микрокальцинаты и опухоли.** Очень мелкие объекты (1-3 мм) на изображении высокого разрешения (4K+). Multi-scale FPN критичен.
- **Дерматоскопия.** Меланомы и невусы. Как раз связано с заданием 16.1 (сегментация PH2), но в виде детекции — для скрининговых задач.
- **Гистопатология.** Поиск метастатических клеток, аномальных митозов на whole-slide images. Изображения гигантские (100K×100K пикселей), детекция работает по тайлам.

Особенности:
- **Сверхвысокое требование к recall.** Пропустить опухоль — летально. Precision можно жертвовать — врач отсеет false positives.
- **Объясняемость.** Регулятор (FDA, Росздравнадзор) требует, чтобы модель показывала, почему она поставила ббокс. Используют Grad-CAM, attention maps. Простые детекторы вне регуляторики не пускают.
- **Pretrained ImageNet помогает мало.** Backbone дообучают долго или используют RadImageNet weights. Self-supervised pre-train (SimCLR, DINO) на собственных снимках даёт прирост.
- **Малые датасеты.** Размеченных снимков с патологиями — единицы тысяч, разметка дорогая (час работы радиолога на снимок). Heavy augmentations + transfer learning.

Архитектурные тренды: 3D-детекторы для КТ/МРТ (RetinaNet 3D), DETR-семейство для гистопатологии (длинные range dependencies). Из метрик — FROC (Free-Response ROC) важнее mAP, потому что считает sensitivity при допустимом числе false positives на снимок.

### OCR (детекция текстовых регионов перед распознаванием)

OCR pipeline почти всегда двухэтапный: сначала детектор находит регионы текста (text detection), потом recognizer распознаёт содержимое (text recognition). Без качественного детектора весь pipeline разваливается.

Сценарии:
- **Документы.** Паспорта, счета, накладные, медкарты. Регионы текста — поля заполнения, таблицы, штрихкоды. Российские примеры: ABBYY FlexiCapture, Контур.Документы, Smart Engines.
- **Уличные сцены.** Распознавание номерных знаков (ANPR), вывесок, дорожных указателей. Текст под произвольным углом, перспективные искажения, частичные occlusions.
- **Рукописный текст.** Школьные тетради, медицинские записи врачей. Самая сложная категория — детектор должен находить «следы» рукописи.

Особенности:
- **Rotated bboxes.** Текст редко идёт строго горизонтально (фотография под углом, изогнутый текст на упаковке). Используют oriented bbox detectors: EAST, DBNet, PSENet.
- **Long aspect ratios.** Текстовая строка может быть 1×30 (символ ширины 1, длина строки 30). Anchor-based детекторы тут плохо работают, anchor-free (FCOS, CenterNet) лучше.
- **Multi-scale обязателен.** Заголовок 100px, мелкий шрифт 15px на одной странице. FPN+PANet с 4-5 уровнями.
- **Очень высокое требование к recall.** Пропущенное слово ломает понимание документа. Используют SoftNMS вместо обычного NMS.

Связь с заданием 19.1: backbone+FPN+anchor-free head — архитектурная база большинства современных text detectors. DIoU/CIoU loss применяются в их обучении (CIoU особенно полезен из-за длинных aspect ratios).

### Аннотация датасетов (semi-supervised pre-labeling)

Размечать данные — самая дорогая часть ML-проекта. Object detection: разметка одного изображения занимает 30 секунд - 5 минут (зависит от количества объектов). Датасет на 100K изображений — это 1000-10000 человеко-часов.

Pre-labeling (auto-annotation) с помощью существующей модели сокращает это в 5-10 раз:
1. Берём pretrained-модель (YOLO, COCO-trained), прогоняем по новому датасету.
2. Получаем automatic predictions.
3. Аннотатор в инструменте (Label Studio, CVAT, Roboflow) только корректирует ошибки и подтверждает правильные.
4. Скорость разметки ускоряется в 5-10x.

Особенности:
- **Domain gap.** COCO-pretrained модель плохо работает на специализированных доменах (медицина, сателлиты, промышленность). Нужно сначала обучить «слабую» модель на 500-1000 ручных примеров, потом её использовать для pre-labeling.
- **Active learning loop.** Pre-label → коррекция → дообучение → pre-label. Каждая итерация улучшает модель и ускоряет разметку. Обычно за 3-5 итераций модель достигает production-качества на 5x меньшем human-effort.
- **Confidence threshold.** Для pre-labeling threshold должен быть низким (0.1-0.3), чтобы модель показывала все «возможные» объекты. Аннотатор удалит лишнее.

Это базовая практика в любой production ML-команде. Российские примеры: Toloka.AI как платформа разметки, Yandex.Cloud DataSphere с auto-annotation модулем.

### Поиск по изображениям и AR-фильтры

**Поиск по изображениям.** Детектор находит объекты на запросе, для каждого вычисляется embedding (через крупный backbone), потом nearest neighbor search в индексе (FAISS, Milvus, Qdrant). Детектор отсекает фон и фокусирует поиск на ключевых объектах.

Примеры:
- **Pinterest Visual Search.** Детектирует одежду на фото пользователя, ищет похожие товары в каталоге.
- **Wildberries Visual Search.** Аналогично — пользователь фотографирует товар, система ищет в каталоге.
- **Google Lens.** Детектирует растения, животных, тексты, товары — каждое разному pipeline.

**AR-фильтры (Snapchat, TikTok, VK Clips, Avatarify).** Детектор находит лицо/руки/тело, после чего на ббокс «вешается» виртуальный объект (маска, шляпа, эффект). Особенности:
- **Real-time на мобильном устройстве.** TFLite/CoreML/NCNN. Детектор должен работать 30-60 fps на iPhone/Android.
- **Стабильность по времени.** Если ббокс «дрожит» от кадра к кадру — фильтр выглядит ужасно. Используют temporal smoothing или tracker.
- **Очень мелкие объекты.** Глаза, губы — 20-50 пикселей. Multi-scale обязателен.

Архитектуры — обычно квантизованные MobileNetV3+SSD или специализированные (BlazeFace, MediaPipe). Backbone маленький, head — single-scale, но оптимизирован под мобильное железо.

## 3. Сравнение с альтернативами: Detection vs Classification vs Segmentation

| Сценарий | Один объект на изображении? | Точные пиксельные маски? | Несколько объектов с bbox+класс? | Решение |
|---|:---:|:---:|:---:|---|
| Один продукт на фото для каталога | Да | — | — | **Classification** (ResNet/EfficientNet) |
| Цвет кожи на фото для медицины | — | Да (точная маска) | — | **Semantic Segmentation** (UNet, DeepLab) |
| Подсчёт людей в магазине | — | — | Да | **Object Detection** (YOLO, Faster R-CNN) |
| Дефект на детали (1 шт. за раз) | Да (если 1 деталь) | Иногда | — | Classification + GradCAM, или Detection если позиций много |
| Гистопатология (множество клеток) | — | Да (для морфометрии) | Да (для подсчёта) | **Instance Segmentation** (Mask R-CNN) |
| Распознавание номеров машин | — | — | Да (одна машина — один bbox с текстом) | Detection + OCR |
| Маркировка объектов на сателлите | — | Иногда | Да | **Detection** (rotated bbox для зданий) |
| Селфи-фильтры с сегментацией волос | — | Да | — | **Segmentation** на лице/волосах |
| Детекция повреждений автомобиля для страховки | — | Иногда (для оценки площади) | Да | **Instance Segmentation** или Detection + классификатор |

**Принципы выбора:**

1. **Один объект, один класс/много классов** → Classification. Самая дешёвая задача.
2. **Один или несколько объектов, нужна локализация** → Detection. Universal-tool.
3. **Нужна площадь / форма / точные границы** → Semantic или Instance Segmentation. Дороже в разметке (5-20x) и обучении.
4. **Нужны точные границы И различение экземпляров** → Instance Segmentation (Mask R-CNN, YOLACT, SOLOv2). Самое дорогое.

Часто Detection — компромисс между Classification (мало информации, дёшево) и Segmentation (много информации, дорого). Для большинства бизнес-задач bbox достаточно.

## 4. Когда использовать готовые решения vs обучать с нуля

В 95% случаев object detection в индустрии — это **fine-tune готовой модели**, а не обучение с нуля. Вопрос только — какую модель и каким инструментом.

### Ultralytics YOLO11 (и YOLOv8)

**Когда выбирать.** Стандартные детекционные задачи на «обычных» данных (фото реального мира, COCO-like распределения), нужен быстрый старт и хорошие метрики из коробки.

Плюсы:
- Установка `pip install ultralytics`, обучение в одну команду.
- Pretrained веса на COCO, легко fine-tune на свой датасет.
- Готовые экспорты в ONNX, TensorRT, CoreML, TFLite.
- Активное community, обновления каждые 2-3 месяца.
- Хорошие метрики на стандартных бенчмарках.

Минусы:
- Лицензия AGPL-3.0 — для коммерческого использования нужна enterprise license от Ultralytics. Это блокер для многих компаний.
- Архитектура — «чёрный ящик» в коде. Кастомизация сложна — много магии в конфигах.

Use case: стартап делает MVP для retail-аналитики на 6 месяцев — Ultralytics YOLO11 даст рабочий прототип за неделю. Для серьёзного long-term production без AGPL — другие варианты.

### torchvision Faster R-CNN / RetinaNet / FCOS

**Когда выбирать.** Исследовательские задачи, кастомизация архитектуры, обучение «с понятным кодом», без лицензионных ограничений.

Плюсы:
- BSD-лицензия torchvision — без ограничений.
- Чистый PyTorch код, легко модифицировать под свою задачу (добавить FPN-уровни, поменять backbone, изменить loss).
- Хорошая документация и интеграция со всем PyTorch-стеком.
- Pretrained на COCO для популярных архитектур.

Минусы:
- Метрики «из коробки» хуже, чем у YOLOv8/v11. Нужен ручной тюнинг.
- Меньше готовых утилит (трекеры, экспорты, deploy-tools).
- Старее: Faster R-CNN — 2015, RetinaNet — 2017, FCOS — 2019. Свежих SOTA-моделей в torchvision нет.

Use case: научная работа, prototype под конкретный домен, внутренний production-проект, где важен полный контроль над архитектурой.

### MMDetection

**Когда выбирать.** Enterprise-level пайплайны, сравнение десятков архитектур, исследовательские проекты с нестандартными конфигами.

Плюсы:
- 50+ архитектур из коробки (Faster R-CNN, RetinaNet, FCOS, ATSS, DETR, YOLOX, Mask R-CNN, CenterNet, ...).
- Конфиги в стиле «наследование» (легко переиспользовать рецепты).
- Распределённое обучение, mixed precision, deterministic mode — всё из коробки.
- Apache 2.0 лицензия — для бизнеса ок.
- Активная поддержка от OpenMMLab (Китай), новые архитектуры добавляются быстро.

Минусы:
- Высокий порог входа. Конфиг-файлы громоздкие, без опыта MMDetection поначалу путает.
- Зависимости тяжёлые (mmcv, специфичные CUDA операторы).
- Документация хороша, но англо-китайская — местами «китайский английский».

Use case: средний/крупный production, R&D-команда сравнивает архитектуры, проект на >6 месяцев. Стандарт в больших компаниях с автономным транспортом, медицинскими приложениями, заводской автоматизацией.

### Detectron2

**Когда выбирать.** Production от Meta-стека или совместимый домен, instance segmentation, panoptic segmentation.

Плюсы:
- Production-ready от Meta (Facebook AI Research).
- Mask R-CNN, Cascade R-CNN, Panoptic FPN — лучшие реализации.
- Apache 2.0 лицензия.
- Хорошо интегрирован с Meta-инструментами (PyTorch native).
- Высокое качество кода и тестов.

Минусы:
- Меньше «свежих» архитектур, чем в MMDetection.
- Установка через `pip` иногда конфликтная (CUDA, PyTorch версии).
- Сложнее для новичков, чем YOLO Ultralytics.

Use case: instance segmentation в production (например, медицинская аналитика, AR-приложения), команды с опытом Meta-стека, R&D с упором на пиксельную точность.

### От нуля (как в задании 19.1)

**Когда оправдано.**

1. **Учебные цели.** Понять, как работает детектор изнутри. Задание 19.1 — именно это. В реальном проекте никто не реализует TAL руками с нуля — берут MMDetection.
2. **Очень специфичные задачи.** Rotated bbox для аэрофото с собственной геометрией, 3D-детекция в медицине, специальные структуры (skeleton-based). Готовых моделей под это нет.
3. **Строгие ограничения по железу.** Embedded-устройство с 1 МБ RAM. Готовые модели не лезут — пишется кастомная архитектура.
4. **Тренировка inference-движка собственной разработки.** Если компания делает свой ML-фреймворк (Yandex CatBoost, Sber GigaChat-style), детектор пишется внутри.

**Когда НЕ оправдано.**

- «Хочу понять, как работает» — для понимания достаточно прочитать код torchvision или туториалы. Реализация с нуля займёт недели и не даст лучших метрик.
- «Хочу сэкономить на лицензии Ultralytics» — берите torchvision или MMDetection, оба BSD/Apache.
- «Хочу полный контроль» — MMDetection даёт полный контроль через конфиги, без необходимости писать с нуля.

**Главный риск самописных детекторов** — баги в loss, label assignment или NMS, которые сложно отловить. В готовых решениях они унесли годы багфиксов. Самописная версия будет давать худшие метрики на 3-5 mAP пунктов из-за мелких неточностей.

## 5. Чек-лист практика

| Задача | Рекомендуемый подход |
|---|---|
| MVP для retail-аналитики (подсчёт людей) | Ultralytics YOLO11n + ByteTrack, fine-tune на 1-2K кадрах из своих камер |
| Production видеонаблюдения с long-term поддержкой | MMDetection (RetinaNet/YOLOX) на ResNet50+FPN, обучение на собственном датасете |
| Контроль качества на конвейере (1-2 класса дефектов) | YOLOv8s или torchvision RetinaNet, RGB+ColorJitter+мелкий MotionBlur, recall > 0.99 |
| ADAS / автономный транспорт | MMDetection или собственный pipeline на базе CenterTrack/BEVFormer + sensor fusion |
| Медицинская детекция патологий | Detectron2 (Mask R-CNN) или MMDetection с RadImageNet pretrained, 3D расширения для КТ |
| OCR text detection | Готовые решения: PaddleOCR, EasyOCR, или DBNet/EAST в MMDetection |
| Pre-labeling датасета | YOLO11 на COCO + Active Learning loop в Label Studio/CVAT |
| AR-фильтры на мобильном | MediaPipe или квантизованный MobileNetV3+SSD через TFLite |
| Pinterest-style visual search | Detection (YOLO/Faster R-CNN) + Embedding (CLIP) + FAISS index |
| Маленький датасет (<500 изображений) | torchvision Faster R-CNN с pretrained COCO + heavy augmentations + early stopping |
| Сильно несбалансированные классы | YOLOv8 с Mosaic+CutMix+Copy-Paste, focal loss или TAL assigner |
| Crowded scenes (толпы, плотный трафик) | MMDetection RetinaNet + SoftNMS, обязательно multi-scale FPN |
| Реал-тайм с edge-устройства (Jetson Nano, Coral) | YOLOv8n квантизованный INT8 через TensorRT или TFLite |
| Точные ббоксы для AR/медицины | CIoU loss + Decoupled Head + аугментации без RandomRotation на больших углах |
| Ансамбль моделей (Kaggle / hi-quality production) | 3-5 моделей разных архитектур + Weighted Boxes Fusion на постпроцессинге |
| Учебная задача / понимание архитектуры | Реализация с нуля по семинару (как 19.1): Backbone+FPN+Decoupled Head+TAL+DIoU+NMS |
| Domain shift далёк от ImageNet | Self-supervised pre-train (DINO, MAE) на собственных неразмеченных + fine-tune детектора |
| Мониторинг качества модели в проде | Логировать per-class AP, distribution of confidence scores, drift detection по входным изображениям |
