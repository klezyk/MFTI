# Прикладное применение: Churn Prediction и методы бинарной классификации

> Этот документ дополняет решение задания 8.1 практической перспективой: где и как методы из задания (Churn prediction, ROC-AUC, LogisticRegression, CatBoost, one-hot encoding, StandardScaler, кросс-валидация, обработка дисбаланса) применяются в реальных задачах IT, аналитики данных, финансов и поиска аномалий.

---

## 1. Churn Prediction — прогноз оттока клиентов

### Где применяется

**Телеком и подписочные сервисы.** Классическая задача — именно она решалась в задании. Оператор связи или SaaS-платформа хочет заранее знать, какие клиенты уйдут в ближайший месяц. Признаки: длительность подписки (ClientPeriod), ежемесячные траты (MonthlySpending), тип контракта, подключённые услуги. На основе вероятности оттока retention-команда расставляет приоритеты: клиентам с вероятностью churn > 0.7 звонят менеджеры, предлагая скидку или бонус. Экономика простая — удержать существующего клиента в 5-7 раз дешевле, чем привлечь нового.

**Банковские продукты.** Банк предсказывает, закроет ли клиент депозит или кредитную карту. Признаки аналогичны: количество транзакций за последние 30/60/90 дней, баланс, количество обращений в поддержку, использование мобильного приложения. Модель ранжирует клиентов по вероятности ухода, после чего CRM-система автоматически запускает цепочку удерживающих коммуникаций: push-уведомления, email, звонок менеджера.

**Онлайн-игры и мобильные приложения.** Предсказание, перестанет ли пользователь играть (DAU churn). Признаки: количество сессий, продолжительность игры, покупки, прогресс по уровням, социальные взаимодействия. Модель срабатывает, когда игрок «остывает» — система отправляет персонализированное предложение (бесплатные ресурсы, ивент) именно в момент наибольшего риска потери.

### Где НЕ стоит применять

- **Когда нет исторических данных об оттоке.** Стартап запустился месяц назад — нет примеров ушедших клиентов для обучения. Здесь лучше использовать эвристики (RFM-анализ) или unsupervised-методы.
- **Когда отток определяется внешними факторами.** Если клиенты уходят из-за повышения цен конкурентом или изменения законодательства, модель на исторических данных будет бесполезна — паттерн поведения «уходящего» клиента изменился.
- **Когда нет возможности action.** Предсказание бессмысленно, если у компании нет инструментов удержания (нет бюджета на скидки, нет retention-команды, продукт нельзя кастомизировать).

### Типичные ошибки на практике

1. **Слишком поздний сигнал.** Если модель предсказывает churn на основе признака «клиент уже подал заявку на отключение», она будет точной, но бесполезной — клиент уже принял решение. Нужно предсказывать за 30-60 дней до события, используя ранние сигналы (снижение активности, рост обращений в поддержку).
2. **Data leakage через TotalSpent.** В задании TotalSpent = ClientPeriod * MonthlySpending (примерно). Если клиент только что пришёл, его TotalSpent низкий — но это не причина churn, а следствие короткого срока. В production нужно аккуратно разделять «каузальные» и «следственные» признаки.
3. **Игнорирование cost-sensitive аспекта.** Не все клиенты одинаково ценны. Удерживать клиента с MonthlySpending=20 может быть невыгодно, если стоимость retention-звонка — 500 рублей. На практике вероятность churn умножают на LTV клиента для приоритизации.

## 2. ROC-AUC как метрика оценки

### Где применяется

**Сравнение моделей без привязки к порогу.** В задании ROC-AUC использовалась для сравнения LogisticRegression (0.8451) и CatBoost (0.8586). Главное преимущество — метрика не зависит от порога классификации. Это критично, когда порог будет определён позже бизнесом (например, «блокируем транзакцию, если вероятность фрода > X», где X зависит от аппетита к риску).

**Fraud detection и скоринг.** В банковском скоринге модель оценивается именно по ROC-AUC, потому что порог отсечения определяется отдельно: risk-менеджер выбирает его на основе trade-off между одобрением заявок (прибыль) и уровнем дефолтов (убытки). Модель с AUC=0.85 позволяет гибко настроить порог; модель с AUC=0.60 бесполезна при любом пороге.

**Мониторинг деградации модели.** В production ROC-AUC отслеживается во времени. Если метрика падает с 0.85 до 0.78 за месяц — это сигнал о data drift: распределение данных изменилось, и модель нужно переобучить. Это стандартная практика в MLOps-пайплайнах.

### Где НЕ стоит применять

- **При сильном дисбалансе классов (1:1000 и хуже).** ROC-AUC может быть оптимистично высокой, потому что FPR (false positive rate) делится на огромное количество негативных примеров. В таких случаях лучше использовать PR-AUC (Precision-Recall AUC), которая фокусируется на качестве предсказаний для редкого класса.
- **Когда нужна интерпретация для бизнеса.** «AUC=0.85» ничего не говорит product-менеджеру. Лучше перевести в бизнес-метрику: «модель найдёт 80% уходящих клиентов, ошибочно пометив 30% лояльных» (конкретный порог, precision/recall).
- **Для задач с неравной ценой ошибок.** Если пропустить фрод стоит 100 000 рублей, а ложная блокировка — 500 рублей, ROC-AUC не учитывает эту асимметрию. Нужна метрика, взвешенная по стоимости ошибок.

### Типичные ошибки на практике

1. **Путать AUC с accuracy.** AUC=0.85 не означает, что 85% предсказаний верны. AUC — это вероятность того, что модель присвоит случайному положительному примеру более высокий score, чем случайному отрицательному.
2. **Сравнивать AUC на разных датасетах.** AUC=0.90 на одном датасете и AUC=0.80 на другом не означает, что первая модель лучше — задачи могут быть разной сложности.
3. **Оптимизировать AUC, когда бизнесу нужен конкретный порог.** Если retention-команда может обработать только 100 клиентов в день, важна precision@100 (точность в топ-100 по score), а не общий AUC.

## 3. LogisticRegression с подбором гиперпараметра C

### Где применяется

**Быстрый baseline для любой задачи классификации.** В задании LogisticRegressionCV с перебором C=[100, 10, 1, 0.1, 0.01, 0.001] дала ROC-AUC=0.8451 за секунды. Это первая модель, которую стоит попробовать: она показывает «потолок» линейных моделей и задаёт нижнюю планку для более сложных методов. Если CatBoost даёт +1% к baseline — возможно, не стоит усложнять production-систему.

**Регулируемые отрасли.** В банковском скоринге и страховании регулятор (ЦБ, Basel Committee, GDPR Article 22) требует объяснить решение модели. Логрегрессия с параметром C=10 означает умеренную регуляризацию — коэффициенты стабильны и интерпретируемы. Каждый коэффициент — вклад конкретного признака: «HasContractPhone=Month-to-month увеличивает вероятность churn на X%».

**Real-time системы с жёсткими SLA.** Inference логрегрессии — одно матричное умножение (микросекунды). Для платёжных шлюзов, обрабатывающих тысячи транзакций в секунду, это принципиально. CatBoost с 1000 деревьями потребует на порядок больше времени.

### Где НЕ стоит применять

- **Сложные нелинейные зависимости.** Если churn зависит от комбинации «короткий срок подписки + высокие траты + отсутствие техподдержки», логрегрессия не поймает это взаимодействие без ручного feature engineering.
- **Категориальные признаки с высокой кардинальностью.** В задании 16 категориальных признаков после one-hot дали 27 колонок — терпимо. Но если бы был признак «город» (1000 значений), one-hot создал бы 999 колонок, и логрегрессия начала бы переобучаться.

### Типичные ошибки на практике

1. **Не подбирать C.** В задании разница между C=0.001 (AUC=0.8329) и C=10 (AUC=0.8451) — существенные 1.2%. В production это может означать тысячи неправильно классифицированных клиентов.
2. **Не стандартизировать данные перед логрегрессией.** Без StandardScaler признаки с разным масштабом (ClientPeriod ~ 1-72, MonthlySpending ~ 18-118, TotalSpent ~ 18-8684) получат несопоставимые градиенты, и модель будет оптимизироваться неравномерно.
3. **Использовать C из обучения на production-данных другого домена.** Оптимальный C=10 для телеком-churn не обязательно оптимален для банковского churn — нужно подбирать заново.

## 4. CatBoost (градиентный бустинг)

### Где применяется

**Табличные данные — золотой стандарт.** В задании CatBoost (n_estimators=1000, learning_rate=0.01) дал ROC-AUC=0.8586, обойдя логрегрессию на 1.35%. На Kaggle и в production gradient boosting (CatBoost, LightGBM, XGBoost) — доминирующий подход для табличных данных. Главное преимущество CatBoost — нативная обработка категориальных признаков через cat_features, без необходимости ручного one-hot encoding.

**Рекомендательные системы.** Ранжирование товаров, фильмов, статей. CatBoost используется как второй этап: сначала candidate generation (быстрые методы), затем CatBoost ранжирует top-100 кандидатов по вероятности клика/покупки. Признаки: история пользователя, характеристики товара, контекст (время дня, устройство).

**Кредитный скоринг (когда регулятор позволяет).** Некоторые банки используют gradient boosting с SHAP-интерпретацией для объяснения решений. SHAP показывает вклад каждого признака в конкретное предсказание, что частично удовлетворяет регуляторные требования.

### Где НЕ стоит применять

- **Очень маленькие датасеты (<500 строк).** Gradient boosting склонен к переобучению. В задании 5282 строки — достаточно. Но на 200 строках логрегрессия будет стабильнее.
- **Потоковые данные с онлайн-обучением.** CatBoost обучается batch-методом — нельзя «дообучить» модель на одном новом примере без переобучения с нуля. Для онлайн-обучения лучше SGDClassifier.
- **Когда inference time критичен.** 1000 деревьев = 1000 последовательных проходов по дереву. Для real-time систем с latency <1 мс это может быть слишком медленно.

### Типичные ошибки на практике

1. **Не контролировать переобучение.** В задании видно: при learning_rate=0.2 и n_estimators=1000 AUC падает до 0.8096 — модель переобучилась. Правило: чем больше деревьев, тем меньше learning_rate. Но при learning_rate=0.01 обучение в 20 раз дольше.
2. **Игнорировать early stopping.** CatBoost поддерживает early_stopping_rounds: обучение останавливается, когда метрика на validation перестаёт улучшаться. Это экономит время и предотвращает переобучение. В задании подбор шёл перебором — в production лучше использовать early stopping.
3. **Обучать финальную модель только на train.** В задании правильно: финальная модель обучена на всём train (5282 строки). Распространённая ошибка — оставить 20% данных на validation и отправить предсказания модели, обученной на 80%. Эти 20% данных потеряны зря.

## 5. One-Hot Encoding

### Где применяется

**Подготовка категориальных признаков для линейных моделей.** В задании 16 категориальных признаков были преобразованы в 27 бинарных через pd.get_dummies(drop_first=True). Линейные модели, SVM, нейросети не умеют работать с категориями напрямую — им нужны числа. One-hot — самый простой и надёжный способ.

**Feature engineering для рекомендательных систем.** Признак «категория товара» (100 значений) — 99 бинарных колонок. Каждая колонка — «является ли товар категорией X?». Это позволяет модели выучить независимый вес для каждой категории.

**Анализ воронки конверсии.** Признак «источник трафика» (organic, paid, social, email, referral) преобразуется в one-hot для регрессионной модели, предсказывающей конверсию. Коэффициенты показывают, какой канал эффективнее.

### Где НЕ стоит применять

- **Высокая кардинальность (>50 уникальных значений).** Признак «город» (1000 значений) создаст 999 колонок. Это замедлит обучение и может привести к переобучению. Альтернативы: target encoding, frequency encoding, embeddings.
- **Древесные модели (CatBoost, LightGBM).** Именно поэтому в задании для CatBoost one-hot не использовался — CatBoost работает с категориями нативно и часто делает это лучше, чем one-hot (используя ordered target statistics внутри).
- **Порядковые признаки.** Если «образование» = {среднее, бакалавр, магистр, PhD}, one-hot потеряет порядок. Лучше OrdinalEncoder: 0, 1, 2, 3.

### Типичные ошибки на практике

1. **Забыть drop_first=True.** Без удаления первой категории возникает мультиколлинеарность: сумма всех one-hot столбцов = 1. Для линейных моделей с регуляризацией это не критично, но для OLS приводит к нестабильным коэффициентам.
2. **Несовпадение категорий train и test.** Если в test есть категория «Satellite» в HasInternetService, которой не было в train, pd.get_dummies создаст лишний столбец. Решение: использовать sklearn.preprocessing.OneHotEncoder с handle_unknown='ignore'.
3. **Применять one-hot к числовым признакам.** IsSeniorCitizen (0/1) уже числовой — one-hot создаст лишний столбец. В задании его корректно обработали как категориальный (astype(str)), но для бинарных признаков это избыточно.

## 6. StandardScaler — стандартизация признаков

### Где применяется

**Обязательная предобработка для линейных моделей.** В задании StandardScaler применялся к трём числовым признакам (ClientPeriod, MonthlySpending, TotalSpent) перед LogisticRegression. Без стандартизации признак TotalSpent (диапазон 18-8684) доминировал бы в градиенте, а ClientPeriod (1-72) практически игнорировался.

**Кластеризация (k-means, DBSCAN).** Алгоритмы, основанные на расстоянии, требуют одинакового масштаба признаков. Без стандартизации кластеры формируются по признаку с наибольшим разбросом.

**Нейронные сети.** Стандартизация входных данных ускоряет сходимость SGD. Это стандартная практика в deep learning — batch normalization по сути делает то же самое внутри сети.

### Где НЕ стоит применять

- **Древесные модели.** Random Forest, XGBoost, CatBoost работают с порядком значений, а не с абсолютными величинами. Стандартизация не повлияет на результат, но добавит ненужный шаг в pipeline.
- **Разреженные данные (TF-IDF, bag-of-words).** StandardScaler вычитает среднее, превращая разреженную матрицу в плотную. Это увеличивает потребление памяти в сотни раз. Для текста лучше MaxAbsScaler или нормализация по строкам.

### Типичные ошибки на практике

1. **Fit на всём датасете, а не только на train.** В задании scaler обучался на train, затем применялся к test с теми же параметрами (mean, std). Если обучить scaler на train+test — это data leakage: тестовая статистика «утекает» в обучение.
2. **Забыть сохранить scaler при деплое модели.** В production нужно сохранить scaler вместе с моделью (через joblib). Если scaler потерян — новые данные будут поданы в модель нестандартизированными, и предсказания окажутся мусором.
3. **Применить scaler к бинарным признакам.** IsSeniorCitizen (0/1) после StandardScaler станет (-0.43, 2.31) — это ухудшит интерпретируемость без выигрыша в качестве. Стандартизировать стоит только непрерывные числовые признаки.

## 7. Кросс-валидация и подбор гиперпараметров

### Где применяется

**Надёжная оценка модели.** В задании LogisticRegressionCV использовала 5-fold CV для подбора C. Это даёт не точечную оценку (AUC=0.85), а распределение (AUC=0.845 +/- 0.01). В production эта информация критична: если разброс по фолдам большой (0.75-0.92), модель нестабильна, и нужно либо собрать больше данных, либо упростить модель.

**Подбор гиперпараметров без переобучения.** В задании для CatBoost перебирались 5 значений n_estimators и 5 значений learning_rate (25 комбинаций). Без отдельного validation set можно случайно выбрать параметры, оптимальные для конкретного split, но не для новых данных. CV решает эту проблему.

**A/B-тестирование ML-моделей.** Перед деплоем новой модели в production её сравнивают со старой на исторических данных через CV. Если новая модель стабильно лучше по всем фолдам — её выкатывают. Если лучше в среднем, но хуже на отдельных фолдах — нужно разобраться.

### Где НЕ стоит применять

- **Временные ряды.** Обычный k-fold перемешивает данные — модель может «увидеть» будущее. Для временных рядов нужна TimeSeriesSplit: обучение на прошлом, тест на будущем.
- **Очень большие данные (>10M строк).** 5-fold CV означает 5 полных обучений модели. Для CatBoost с 1000 деревьями на 10M строках это часы вычислений. Лучше один train/valid split с достаточно большой validation выборкой.

### Типичные ошибки на практике

1. **Подбирать гиперпараметры на test set.** Если вы подбираете C на test, а потом оцениваете модель на том же test — метрика завышена. Правильная схема: train для обучения, validation (или CV) для подбора параметров, test — только для финальной оценки.
2. **Не использовать стратификацию при дисбалансе.** В задании churn — 26% положительных примеров. При обычном k-fold один фолд может получить 20% churn, другой — 32%. Стратифицированная CV (StratifiedKFold) сохраняет пропорции в каждом фолде.
3. **Переоптимизация на CV.** Перебрать 10 000 комбинаций гиперпараметров на CV и выбрать лучшую — это переобучение на валидационные фолды. На практике достаточно 20-50 комбинаций (RandomizedSearchCV вместо GridSearchCV).

## 8. Обработка дисбаланса классов

### Где применяется

**Churn prediction.** В задании дисбаланс умеренный — 74% vs 26%. Модель без коррекции уже работает неплохо (ROC-AUC=0.86). Но при более сильном дисбалансе (5% churn) модель может просто предсказывать «не уйдёт» для всех — accuracy=95%, но ноль практической пользы.

**Fraud detection.** Фрод составляет 0.1-1% транзакций. Без коррекции дисбаланса модель игнорирует редкий класс. Методы: oversampling (SMOTE), undersampling, class_weight='balanced' в sklearn, scale_pos_weight в CatBoost.

**Медицинская диагностика.** Редкие заболевания — 0.01% популяции. Пропустить больного (false negative) критически опасно, поэтому модель должна быть чувствительна к редкому классу, даже ценой false positives.

### Где НЕ стоит применять

- **Сбалансированные данные (40-60%).** Коррекция дисбаланса на сбалансированных данных может ухудшить модель — она начнёт переоценивать один из классов.
- **Когда ROC-AUC — основная метрика.** ROC-AUC инвариантна к балансу классов (она работает с рангами, а не с абсолютными вероятностями). В задании именно поэтому модели показали хороший результат без специальной обработки дисбаланса.

### Типичные ошибки на практике

1. **Применять SMOTE до разбиения на train/test.** Синтетические примеры, созданные из тестовых данных, попадут в train — это data leakage. SMOTE применяется только внутри train-фолда.
2. **Путать баланс классов и калибровку вероятностей.** class_weight='balanced' улучшает recall для редкого класса, но делает вероятности некалиброванными: модель завышает вероятность churn. Если бизнес использует вероятности для принятия решений (а не просто ранжирование), нужна пост-калибровка (CalibratedClassifierCV).
3. **Слепо oversampling-ить.** SMOTE создаёт синтетические примеры интерполяцией между соседями. Если пространство признаков неоднородно (кластеры), синтетические примеры могут попасть в «пустые» области и сбить модель.

## Сравнение подходов: LogisticRegression vs CatBoost

| Критерий | LogisticRegression | CatBoost |
|----------|:------------------:|:--------:|
| **Качество в задании** | ROC-AUC = 0.8451 | ROC-AUC = 0.8586 |
| **Скорость обучения** | Секунды | Минуты |
| **Скорость inference** | Микросекунды | Миллисекунды |
| **Интерпретируемость** | Высокая (коэффициенты) | Средняя (SHAP, feature importance) |
| **Предобработка** | StandardScaler + one-hot (обязательно) | Минимальная (cat_features) |
| **Нелинейные зависимости** | Не ловит | Ловит автоматически |
| **Устойчивость к переобучению** | Высокая (с регуляризацией) | Требует настройки (lr, n_estimators) |
| **Регуляторные требования** | Полностью соответствует | Требует дополнительной интерпретации |
| **Малый датасет (<500)** | Стабильна | Склонна к переобучению |
| **Онлайн-обучение** | SGDClassifier — да | Нет (только batch) |

### Decision framework: что выбрать

```
1. Нужна ли объяснимость для регулятора?
   -> Да -> LogisticRegression

2. Inference <1 мс?
   -> Да -> LogisticRegression

3. Данных < 1000?
   -> Да -> LogisticRegression с L2

4. Есть категориальные признаки с >20 уникальными значениями?
   -> Да -> CatBoost (нативная обработка)

5. Нужно максимальное качество, ограничений нет?
   -> CatBoost / LightGBM + подбор гиперпараметров

6. Первая итерация, нужен baseline?
   -> LogisticRegression -> затем CatBoost для сравнения
```

## Чек-лист практика

| Задача | Рекомендуемый подход |
|--------|---------------------|
| Churn prediction (телеком, SaaS) | CatBoost с cat_features, ROC-AUC для оценки, LogisticRegression как baseline |
| Fraud detection (real-time) | LogisticRegression (скорость), порог 0.1-0.3, PR-AUC для мониторинга |
| Fraud detection (batch) | CatBoost/LightGBM, SMOTE внутри CV, ранжирование по score |
| Кредитный скоринг (регулируемый) | LogisticRegression + L2, Pipeline (StandardScaler + OneHotEncoder), CV-5 |
| Рекомендательная система (ранжирование) | CatBoost с cat_features, feature importance для отладки |
| Быстрый baseline на табличных данных | LogisticRegressionCV (автоматический подбор C) — 5 минут работы |
| Соревнование Kaggle (таблица) | CatBoost + LightGBM + stacking, early stopping, Optuna для гиперпараметров |
| Категориальные признаки + линейная модель | One-hot encoding (drop_first=True), StandardScaler для числовых |
| Категориальные признаки + бустинг | Нативная обработка (cat_features в CatBoost), без one-hot |
| Дисбаланс 90/10 + ROC-AUC метрика | Без специальной коррекции — ROC-AUC инвариантна к балансу |
| Дисбаланс 99/1 + precision/recall | class_weight='balanced' или SMOTE (только внутри train) |
| Мониторинг модели в production | Отслеживать ROC-AUC на свежих данных, алерт при падении >3% |
